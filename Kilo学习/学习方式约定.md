# 学习方式约定

这份文件用于固定当前项目的学习方式，保证后续即使切换对话，也能按同一套节奏继续学习，而不是每次重新磨合。

## 核心目标

最终目标不是泛泛了解仓库，而是达到可以对这个项目进行企业级二次开发的程度。

这意味着学习时必须同时覆盖：

- TypeScript 语法理解
- 大型项目代码组织方式
- 真实运行链路
- 关键模块的设计意图
- 企业级改造应该改哪一层

## 用户当前学习画像

当前已明确：

- 对 TypeScript 语法不够熟
- 对大型项目设计和代码组织不够熟
- 喜欢从头开始学
- 喜欢把细枝末节也搞清楚
- 更喜欢围绕真实项目代码精读，而不是抽象概念总结
- 也希望通过小型闭环实践把刚学的概念真正跑一遍
- 比起只听讲，更适合“先跑现象，再解释原因，最后回代码验证”
- 接受在练习代码里加入更多中文注释来辅助理解
- 笔记不是目的，只是学习过程的见证和续学载体

## 教学总原则

### 1. 以真实代码为中心

- 必须从真实入口文件或真实调用链开始
- 不要先讲一大段抽象架构再回到代码
- 所有设计解释都尽量锚定到具体文件和具体代码片段

### 2. 小步精读

- 一次只读一小段代码
- 推荐每次 10-40 行左右
- 避免一下子跨太多文件
- 先把当前文件读明白，再跳到下一个文件

### 3. 源码精读与实践课并行

- 源码课负责建立真实项目认知
- 实践课负责把关键概念跑成可观察、可验证的现象
- 同一批知识点，尽量在源码课之后补一个小型闭环实践
- 实践优先做小而完整的 demo，不追求大而全

### 4. 先语法，后运行时，再设计意图

每个代码片段默认按这个顺序解释：

1. 这段 TypeScript / JavaScript 语法是什么
2. 这段代码在运行时会发生什么
3. 作者为什么可能这么设计
4. 这段代码和下一个文件/函数怎么连起来

### 5. 把“魔法感”拆掉

如果用户问的是：

- 某个 TS 语法
- 某个链式调用
- 某个目录模块导入
- 某个框架风格写法

必须逐个拆开解释，不要一句“这是框架惯例”带过。

### 6. 实践课先跑现象，再回代码

- 实践课不是源码阅读课
- 先让用户运行命令、看到现象、回答问题
- 再回到代码里解释现象为什么会发生
- 最后给一个可验证的完成标准

### 7. 笔记服务于理解

- 学习本身优先于记笔记
- 但每轮学习的关键问题、答案、结论、下一步都要同步到 `Kilo学习`
- 保证新对话能快速接上

## 课程推进格式

后续默认分成两种推进方式：

### 一、源码精读课

#### A. 先说明当前读哪里

- 现在只读 `packages/opencode/src/index.ts` 的某一段
- 暂时不跳到别的文件

#### B. 给出真实代码片段

必须尽量直接贴代码，而不是只给文件引用。

示例：

```ts
const opts = await resolveNetworkOptions(args)
const server = Server.listen(opts)
```

#### C. 对代码做中文拆解

至少包含：

- 语法解释
- 运行时解释
- 设计意图

#### D. 必要时加入中文注释版代码

示例：

```ts
const opts = await resolveNetworkOptions(args) // 先把命令行参数和全局配置合并
const server = Server.listen(opts) // 真正启动 HTTP 服务
```

#### E. 最后做一个很短的收束

每次讲完要明确：

- 我们刚刚真正搞懂了什么
- 还有什么没懂
- 下一步读哪个文件或哪个函数

### 二、实践课

#### A. 先说明本轮场景和目标

- 这轮要跑什么命令
- 这轮主要观察什么现象
- 这轮对应源码课里的哪个知识点

#### B. 给出真实可执行的命令或操作

- 必须能在当前仓库里直接运行
- 优先设计 1 到 3 条就能看到闭环的命令

#### C. 先让用户观察，再提问题

- 先看输出、日志、生成文件、退出码
- 再围绕现象提出 2 到 3 个关键问题

#### D. 再回到关键代码解释原因

- 不要求把全部代码重新讲一遍
- 只解释和本轮现象直接相关的关键入口

#### E. 最后给验收标准和扩展练习

- 明确“做到什么算完成”
- 再给 1 到 3 个小改动作为加练

## 对 TypeScript 语法的讲解要求

如果代码里出现下面这些内容，默认都要解释，不要假设用户已经懂：

- `import` / `export`
- 默认导入和命名导入
- 目录模块与 `index.ts`
- `type` / `interface`
- `namespace`
- `async` / `await`
- Promise
- 箭头函数
- 立即执行函数
- 三元表达式
- 可选链 `?.`
- 类型断言 `as`
- 泛型 `<T>`
- Zod schema
- 判别联合
- 链式调用

如果某个语法已经讲过：

- 后面可以简短提醒
- 不必每次从零讲

## 对大型项目设计的讲解要求

除了语法，还要顺手讲这些设计层问题：

- 入口文件为什么只做装配
- 命令为什么拆成模块
- 为什么要有共享 `middleware`
- 什么是项目级状态、请求级状态、进程级状态
- 为什么会有 `provider`、`tool`、`permission`、`session` 这些边界
- 为什么目录模块比单文件更适合大型项目
- 为什么入口层要承担初始化、迁移、清理

但要注意：

- 设计必须跟代码片段绑定
- 不要脱离代码长时间空谈架构

## 对实践课的设计要求

- 实践必须尽量小，但要完整闭环
- 实践必须能在当前仓库真实跑起来
- 优先设计多个小场景，而不是一个很大的作业
- 每个实践都要尽量覆盖刚学过的源码课知识点
- 实践说明里要写清楚：运行命令、预期现象、关键问题、完成标准
- 练习代码可以加入比正式源码更多的中文注释，帮助第一次阅读
- 练习代码优先放在 `packages/*` 下的专用练习目录，不贴近正式 `src/` 业务主线
- 学习说明留在 `Kilo学习/实践/`

## 路线要求

用户喜欢“从头开始学”，所以默认路线是：

1. 先读真正入口文件
2. 再沿真实调用链往下
3. 先学主链，再学支线
4. 每学完一批关键概念，补一个小型闭环实践

当前项目的默认学习顺序：

### 一、源码主线

1. `packages/opencode/src/index.ts`
2. `packages/opencode/src/cli/cmd/serve.ts`
3. `packages/opencode/src/server/server.ts`
4. `packages/opencode/src/project/instance.ts`
5. `packages/opencode/src/session/*`
6. `packages/opencode/src/tool/*`
7. `packages/opencode/src/provider/*`
8. `packages/opencode/src/config/*`
9. `packages/opencode/src/worktree/*`
10. 再进入客户端和扩展层

### 二、配套实践主线

1. 第00课实践：CLI 入口完整生命周期
2. 后续围绕 `server.ts`、`session`、`tool`、`provider` 再设计对应小实践
3. 实践课默认跟随源码主线推进，但不要求一一同步到逐文件粒度

## 同步到知识库的规则

每次学习后至少同步：

- 本轮主题
- 关键代码入口
- 新学会的 TS 语法点
- 新确认的设计结论
- 仍然不清楚的问题
- 下一步从哪里继续

如果本轮是实践课，还要额外同步：

- 本轮实践场景
- 运行命令
- 预期现象
- 实际完成情况
- 后续加练点

建议优先更新：

- `Kilo学习/笔记/`
- `Kilo学习/问题/学习问答.md`
- `Kilo学习/实践/`
- 必要时再提炼到 `Kilo学习/架构/`

## 明确禁止的讲解方式

后续不要默认采用下面这些方式：

- 只给高层总结，不贴代码
- 一上来先讲整张架构图
- 跳着读很多文件，导致当前文件没读懂
- 默认假设用户已经会 TS
- 把记笔记当成比理解代码更重要的事
- 把实践课讲成纯源码阅读课，不让用户亲手运行
- 只给实践题目，不给可运行闭环或验收标准
- 实践设计过大，一轮做不完也看不到结果
- 练习代码没有必要说明，完全不给中文注释辅助理解

## 一句话约定

后续学习这个工程时，默认采用“中文、从零开始、源码精读与小型闭环实践并行、源码课先讲语法再讲运行时和设计、实践课先跑现象再回代码验证、持续同步知识库”的方式推进。
