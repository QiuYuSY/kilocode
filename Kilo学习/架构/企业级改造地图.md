# 企业级二次开发改造地图

## 这篇文档的目标

前面的文档主要在讲“系统怎么跑”。这一篇专门回答：

如果最终目标是企业级二次开发，不同需求应该优先改哪一层，尽量避免走错路。

## 总原则

### 1. 优先改内核，不优先改客户端

对这个仓库来说：

- 真正的能力内核在 `packages/opencode/`
- `packages/kilo-vscode/` 主要是客户端壳层

所以：

- 改模型、工具、权限、会话，优先改 CLI 内核
- 改产品表现、交互、面板，再看 VS Code 扩展

### 2. 先找现有边界，再决定是否动核心

优先顺序通常是：

1. config
2. plugin hook
3. provider / tool 扩展点
4. route 注入
5. 核心模块改造

### 3. 企业需求尽量落到“稳定层”

更稳定的层通常是：

- provider
- permission
- plugin
- gateway route
- workspace / instance

尽量不要把企业逻辑直接塞进 session 内部细碎分支里。

## 需求到代码落点的映射

### 私有模型网关接入

优先看：

- [provider.ts](../../packages/opencode/src/provider/provider.ts)
- [models.ts](../../packages/opencode/src/provider/models.ts)
- [transform.ts](../../packages/opencode/src/provider/transform.ts)
- [auth.ts](../../packages/opencode/src/provider/auth.ts)

推荐路径：

- 先尝试 `config.provider` + `openai-compatible`
- 需要登录时再加 plugin auth
- 需要协议兼容时再改 transform

不要先改 session 或 tool。

### 组织级模型目录与白名单

优先看：

- [models.ts](../../packages/opencode/src/provider/models.ts)
- [model-cache.ts](../../packages/opencode/src/provider/model-cache.ts)
- [routes.ts](../../packages/kilo-gateway/src/server/routes.ts)

推荐路径：

- 仿照 Kilo / Apertis 的动态注入模式
- 按组织或租户拉远端模型目录
- 配合 cache clear 和 organization 切换

### 企业鉴权与 SSO

优先看：

- [auth.ts](../../packages/opencode/src/provider/auth.ts)
- [index.ts](../../packages/plugin/src/index.ts)
- [plugin.ts](../../packages/kilo-gateway/src/plugin.ts)

推荐路径：

- 优先做 plugin auth
- 让 CLI 保存统一 auth 信息
- 避免每个业务 route 各做一份登录态

### 审计日志与操作留痕

优先看：

- [index.ts](../../packages/plugin/src/index.ts)
- [index.ts](../../packages/opencode/src/plugin/index.ts)
- [processor.ts](../../packages/opencode/src/session/processor.ts)
- [telemetry.ts](../../packages/kilo-telemetry/src/telemetry.ts)

推荐路径：

- 对工具执行，用 `tool.execute.before/after`
- 对模型完成，用 `Telemetry.trackLlmCompletion`
- 对高层会话事件，可补 Bus 订阅或 route 转发

### 权限审批与策略治理

优先看：

- [next.ts](../../packages/opencode/src/permission/next.ts)
- [task.ts](../../packages/opencode/src/tool/task.ts)
- [bash.ts](../../packages/opencode/src/tool/bash.ts)
- [agent.ts](../../packages/opencode/src/agent/agent.ts)

推荐路径：

- 主审批流优先改 `PermissionNext`
- 子 agent 约束优先改 `task.ts`
- shell 风险控制优先改 `bash.ts`

注意：

- plugin 的 `permission.ask` 不覆盖主 tool 链

### 组织级模式与企业工作流

优先看：

- [routes.ts](../../packages/kilo-gateway/src/server/routes.ts)
- [agent.ts](../../packages/opencode/src/agent/agent.ts)
- `packages/kilo-gateway` 的 organization modes 相关 API

推荐路径：

- 把企业模式定义放控制面或网关侧
- CLI 负责拉取和执行

### 多租户、远端工作区、控制平面

优先看：

- [workspace-context.ts](../../packages/opencode/src/control-plane/workspace-context.ts)
- [workspace-router-middleware.ts](../../packages/opencode/src/control-plane/workspace-router-middleware.ts)
- [instance.ts](../../packages/opencode/src/project/instance.ts)

推荐路径：

- 组织级路由用 workspace 边界
- 项目级资源隔离用 instance 边界

### 数据治理与合规

优先看：

- [llm.ts](../../packages/opencode/src/session/llm.ts)
- [transform.ts](../../packages/opencode/src/provider/transform.ts)
- [telemetry.ts](../../packages/kilo-telemetry/src/telemetry.ts)

当前已知有利点：

- telemetry 默认 `recordInputs: false`
- telemetry 默认 `recordOutputs: false`
- 部分 provider 默认 `store: false`

这说明系统已经有一定的数据最小化倾向。

## 企业二开推荐顺序

### 第一步：先做非侵入式接入

- 加 provider
- 加 auth
- 加 header
- 加基础审计

### 第二步：再做治理

- 工具权限
- 审批流
- 子 agent 约束
- 模型白名单

### 第三步：最后再做深层行为改造

- session 调度
- compaction 策略
- agent 策略引擎
- 控制平面深度联动

## 当前最值得警惕的误区

- 把企业逻辑优先塞进 VS Code 扩展，而不是 CLI 内核
- 把权限治理只放在 plugin 层，而没有进入 `PermissionNext`
- 把 provider 协议兼容写进 session 主链
- 忽略组织切换后的 model cache 失效
- 忽略子 agent 的权限传播与越权问题

## 当前建议的下一轮下钻方向

为了更接近企业接管实战，下一轮最值得继续的是：

- `agent + config` 如何形成组织级执行策略
- `workspace + worktree` 如何支撑团队级隔离和并行开发
- `server/routes + sdk` 如何安全地扩展企业 API
