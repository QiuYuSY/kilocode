# Session 执行链路

## 目标

这一篇回答一个核心问题：

用户向 session 发一条消息后，系统如何把这条消息变成一次完整的 agent 执行，包括模型调用、工具调用、状态更新和最终收敛。

## 先看入口

Session 相关 HTTP 路由在 [session.ts](../../packages/opencode/src/server/routes/session.ts)。

其中最关键的入口是：

- `POST /session/:sessionID/message`
- `POST /session/:sessionID/prompt_async`
- `POST /session/:sessionID/command`
- `POST /session/:sessionID/shell`

对普通对话最重要的是：

- `POST /:sessionID/message`
  - 调用 `SessionPrompt.prompt()`
  - 再进入 `SessionPrompt.loop()`

## 链路总览

主链路可以先记成：

`session route`
-> `SessionPrompt.prompt()`
-> `createUserMessage()`
-> `SessionPrompt.loop()`
-> `resolveTools()`
-> `SessionProcessor.create().process()`
-> `LLM.stream()`
-> 模型流式输出 text / reasoning / tool-call / tool-result
-> 持续写回 message / part / status / summary / patch

## 第一层：Session 路由负责协议，不负责执行细节

[session.ts](../../packages/opencode/src/server/routes/session.ts) 的职责主要是：

- 校验参数
- 调用 session 领域方法
- 在需要时开启 stream 返回

它本身并不承担 agent 执行逻辑，真正的“对话引擎”在 `SessionPrompt` 和 `SessionProcessor`。

## 第二层：`SessionPrompt.prompt()` 创建用户消息

[prompt.ts](../../packages/opencode/src/session/prompt.ts) 中的 `prompt()` 做了几件重要事情：

- 获取 session
- 清理 revert 状态
- 创建用户消息和 parts
- `Session.touch()` 更新时间
- 把兼容层的 `tools` 输入转成 permission
- 如果需要回复，就进入 `loop()`

这里可以把它理解成“把 HTTP 输入转成会话内的标准消息结构”。

### `createUserMessage()` 做了什么

它会把本次输入固化成 `MessageV2.User`，其中会写入：

- `agent`
- `model`
- `system`
- `format`
- `variant`
- `editorContext`

这意味着一次用户输入并不只是文本，它同时携带本轮执行策略和上下文。

## 第三层：`SessionPrompt.loop()` 是调度中枢

`loop()` 是 session 执行链里最关键的调度器。

它负责：

- 控制一个 session 是否 busy
- 支持取消与恢复
- 维护一轮 turn 的开始和结束事件
- 持续扫描当前消息流
- 处理 compaction、subtask、普通推理三种分支

### 1. session 的忙闲状态

`SessionPrompt` 内部用 `Instance.state()` 保存每个 sessionID 的 `AbortController` 和回调队列。

这意味着：

- 一个 session 同时只能有一条主执行链
- 取消、恢复、排队都围绕这层状态实现

### 2. `loop()` 每轮先读当前消息流

它会取到当前 session 的消息，识别：

- 最后一个 user message
- 最后一个 assistant message
- 最后一个完成态 assistant
- 当前尚未处理完的 compaction / subtask

这说明 session 不是“每次只看最新请求”，而是在已有消息流上持续推进状态机。

### 3. `loop()` 有三种主分支

#### 分支 A：subtask

如果存在待执行的 `subtask` part，`loop()` 会：

- 创建一条 assistant message
- 直接调用 `TaskTool`
- 把结果写回 tool part
- 必要时再补一条 synthetic user message，引导主流程继续总结或后续处理

#### 分支 B：compaction

如果上下文过长，或者已有待处理 compaction，`loop()` 会交给 `SessionCompaction.process()`。

#### 分支 C：normal processing

普通对话时，会进入：

- 取 agent
- 取 model
- 构造 assistant message
- 解析本轮可用 tools
- 组装 system prompt 和 messages
- 调用 `processor.process()`

## 第四层：`resolveTools()` 负责把平台能力变成模型可调用工具

`prompt.ts` 里的 `resolveTools()` 是一个很重要的桥梁层。

它会整合：

- `ToolRegistry.tools()` 返回的内置工具和插件工具
- `MCP.tools()` 返回的 MCP 工具
- 权限系统 `PermissionNext`
- plugin hook

### `ToolRegistry.tools()` 的意义

在 [registry.ts](../../packages/opencode/src/tool/registry.ts) 中可以看到：

- 内置工具统一注册
- 插件工具会被纳入统一列表
- 不同模型会切换 `apply_patch` 或 `edit/write`

这意味着工具系统已经天然支持：

- 工具扩展
- 工具治理
- 按模型能力适配

### `resolveTools()` 的执行包装

每个工具在注入给模型前，都会被包装出统一行为：

- 统一 tool schema
- 统一上下文对象
- `tool.execute.before`
- `tool.execute.after`
- 统一权限检查入口

这正是企业二开里做审计、白名单、埋点、限流的关键位置。

## 第五层：`SessionProcessor.process()` 负责消费模型流

[processor.ts](../../packages/opencode/src/session/processor.ts) 是真正的执行引擎。

它会消费 `LLM.stream()` 返回的流，并根据不同事件更新 session 状态。

### 它处理的主要流事件

- `start`
- `reasoning-start` / `reasoning-delta` / `reasoning-end`
- `text-start` / `text-delta` / `text-end`
- `tool-input-start`
- `tool-call`
- `tool-result`
- `tool-error`
- `start-step`
- `finish-step`
- `error`

### 它做的核心事情

- 把 reasoning、text、tool 都落成 `MessageV2.Part`
- 把工具 pending/running/completed/error 状态写回
- 记录 token、cost、finish reason
- 在 step 前后追踪 snapshot / patch
- 触发 summary
- 判断是否需要 compaction
- 处理 retry
- 在异常或中断时收尾

可以把它理解成“模型输出事件 -> 会话状态变化”的翻译器。

## 第六层：`LLM.stream()` 负责模型调用抽象

[llm.ts](../../packages/opencode/src/session/llm.ts) 是模型调用的统一入口。

它做的事情包括：

- 根据 `Provider` 解析真实语言模型
- 组装 system prompt
- 合并 agent / model / variant / provider options
- 解析 headers
- 解析 tools
- 调用 AI SDK 的 `streamText()`

### 这里能看到几个重要边界

#### 1. system prompt 不是静态字符串

system prompt 来自多层拼装：

- `SystemPrompt.soul()`
- provider prompt
- agent prompt
- 调用方额外传入的 system
- 用户消息上的 system

这说明 agent 行为调优的主要入口并不只在一个 prompt 文件里，而是一个组合系统。

#### 2. tools 是模型侧能力的一等公民

`streamText()` 直接拿到 tools、toolChoice、activeTools，这说明系统是以“模型可原生发起工具调用”为基本范式设计的。

#### 3. provider header 与 telemetry 已经深度接入

这里会根据 provider 写入额外 header，也会启用 telemetry tracer。企业接入私有网关或做审计时，这里通常是重点改造点之一。

## Agent 在链路中的位置

[agent.ts](../../packages/opencode/src/agent/agent.ts) 说明 agent 不是简单名称，而是一组执行策略：

- prompt
- permission
- model
- options
- steps
- mode

在 `loop()` 中，agent 决定：

- 本轮使用什么 prompt
- 允许什么工具
- 最多跑多少步
- 是否需要进入特定模式

所以以后你要改“agent 的人格、权限、工作方式”，主要就是看这里和 `SessionPrompt` / `LLM` 的交界。

## 当前结论

可以把 session 执行链理解成 4 层：

### 层 1：协议层

- `server/routes/session.ts`
- 接收 HTTP 请求

### 层 2：调度层

- `session/prompt.ts`
- 管一轮对话如何推进

### 层 3：执行层

- `session/processor.ts`
- 管模型事件、工具结果、状态落库

### 层 4：模型抽象层

- `session/llm.ts`
- `provider/*`
- 负责真实模型调用

## 对企业二开的直接启发

- 改 API 行为：先看 `server/routes/session.ts`
- 改对话调度、上下文组织、步数策略：先看 `session/prompt.ts`
- 改工具执行、重试、patch、状态机：先看 `session/processor.ts`
- 改模型接入、headers、provider 行为：先看 `session/llm.ts` 和 `provider/*`
- 改 agent 策略、权限、模式：先看 `agent/agent.ts`
