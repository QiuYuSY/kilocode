# 启动与请求作用域

## 目标

这一篇回答两个关键问题：

- `kilo serve` 是怎么从 CLI 命令变成 HTTP 服务的
- 一个请求进入服务端后，为什么能自动绑定到对应的目录、project、worktree 和 workspace

## 启动链路

### 1. CLI 进程从 `src/index.ts` 开始

入口文件是 [index.ts](../../packages/opencode/src/index.ts)。

这里做了几类全局初始化：

- 初始化日志
- 设置 CLI 相关环境变量
- 初始化 telemetry
- 做一次性数据库迁移
- 注册所有命令
- 在退出时做 telemetry 和 instance 清理

对我们当前最重要的是，它把 `ServeCommand` 注册进了 CLI。

### 2. `serve` 命令进入 `src/cli/cmd/serve.ts`

[serve.ts](../../packages/opencode/src/cli/cmd/serve.ts) 是 `kilo serve` 的实现。

它做的事情很集中：

- 读取并解析网络参数
- 调用 `Server.listen(opts)` 启动服务
- 在本地开发态下启动 workspace 同步
- 监听 `SIGTERM`、`SIGINT`、`SIGHUP`
- 收到信号后执行优雅退出

优雅退出里会做三件事：

- `Instance.disposeAll()`
- `server.stop(true)`
- 停止 workspace 同步

这说明服务端不是“启动后放着不管”，而是明确维护了实例级资源回收。

### 3. 网络参数的真实来源

[network.ts](../../packages/opencode/src/cli/network.ts) 说明 `serve` 的网络配置不是只看命令行参数，还会和全局配置合并。

合并逻辑包括：

- `port`
- `hostname`
- `mdns`
- `mdnsDomain`
- `cors`

这个点对企业部署很重要，因为它说明运行形态是可以通过配置层统一控制的。

### 4. `Server.listen()` 进入 Hono server

[server.ts](../../packages/opencode/src/server/server.ts) 负责真正装配服务端。

它在 `App()` 里集中完成：

- 错误处理
- basic auth
- 请求日志
- CORS
- workspace / directory 请求作用域绑定
- route 挂载
- SSE 事件流

然后在 `listen()` 里调用 `Bun.serve()` 启动 HTTP 服务。

## 请求作用域是怎么绑定的

这是目前最值得吃透的一层，因为它解释了为什么同一个 server 可以服务不同目录、不同 workspace。

### 1. 请求先提取 `workspace` 和 `directory`

在 [server.ts](../../packages/opencode/src/server/server.ts) 中间件里，服务端会从下面位置提取上下文：

- query 参数里的 `workspace`
- header 里的 `x-kilo-workspace`
- query 参数里的 `directory`
- header 里的 `x-kilo-directory`

如果没有显式传 `directory`，就退回到 `process.cwd()`。

### 2. 先建立 `WorkspaceContext`

[workspace-context.ts](../../packages/opencode/src/control-plane/workspace-context.ts) 用一个轻量上下文只保存 `workspaceID`。

这个上下文的职责很单纯：

- 让后续代码知道当前请求属于哪个 workspace
- 供 workspace 路由转发逻辑读取

### 3. 再建立 `Instance` 上下文

[instance.ts](../../packages/opencode/src/project/instance.ts) 是请求作用域的关键。

`Instance.provide()` 会：

- 先把目录标准化
- 按目录做缓存
- 如果这个目录还没有实例，就执行 `boot()`
- 将当前调用绑定到这个目录对应的上下文

`boot()` 会拿到三类核心信息：

- `directory`
- `worktree`
- `project`

如果调用方没有直接提供 `project` 和 `worktree`，它会通过 `Project.fromDirectory()` 推导出来。

这意味着：

- “实例”是按目录维度缓存的
- 状态生命周期不是单个请求级，而是目录级
- 同目录的多个请求会复用同一个 instance 状态

### 4. `InstanceBootstrap()` 初始化目录级资源

[bootstrap.ts](../../packages/opencode/src/project/bootstrap.ts) 会在 instance 首次 boot 时初始化一批目录级能力：

- Plugin
- KiloSessions
- Format
- LSP
- FileWatcher
- File
- Vcs
- Snapshot
- Truncate

这一步很关键，因为它解释了为什么很多能力看起来像“全局单例”，但其实是“按项目目录懒初始化的单例”。

### 5. `Instance.state()` 是项目级懒单例模式

根据 [packages/opencode/AGENTS.md](../../packages/opencode/AGENTS.md) 和 [instance.ts](../../packages/opencode/src/project/instance.ts)，很多模块通过 `Instance.state(init, dispose?)` 注册自己的目录级状态。

这带来几个重要推论：

- 多项目之间的状态天然隔离
- 同一项目内的状态可复用
- 做企业二开时，很多扩展能力应该优先挂在 instance state 上，而不是做真正的进程级全局变量

## Workspace 路由转发

[workspace-router-middleware.ts](../../packages/opencode/src/control-plane/workspace-router-middleware.ts) 展示了另一层边界。

当前逻辑是：

- 只在本地开发态启用
- 如果存在 `workspaceID`，就尝试取到对应 workspace
- 再按 workspace 类型选择 adaptor
- 将请求转发给目标 workspace

这说明仓库已经具备“请求不一定在本地实例内处理，也可能按 workspace 被路由出去”的设计方向。

对企业级二开来说，这很值得关注，因为它天然靠近：

- 远端工作区
- 多环境代理
- 控制平面
- 集中式管理

## 当前结论

可以把这一整套机制理解成两层作用域：

### 层 1：workspace 作用域

- 回答“请求属于哪个逻辑工作区”
- 决定是否需要被转发

### 层 2：instance 作用域

- 回答“请求绑定到哪个本地目录实例”
- 提供 project、worktree、state、watcher、LSP 等目录级能力

## 对企业二开的直接启发

- 如果要加组织级路由或多环境代理，先看 workspace 相关链路
- 如果要加项目级缓存、审计、策略、能力开关，先看 instance 相关链路
- 如果要做稳定扩展，尽量顺着现有上下文模型扩展，不要另起一套并行状态系统
