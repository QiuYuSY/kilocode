# Tool、Permission 与治理链路

## 为什么这块是企业二开的主战场

企业在二开时最关心的通常不是“能不能调模型”，而是：

- 工具能不能被限制
- 哪些操作要审批
- 能不能做审计留痕
- 能不能做白名单、黑名单、组织策略
- 子任务和子 agent 会不会越权

这些问题主要都落在 tool、permission、plugin 三层。

## 关键代码入口

- Tool 抽象: [tool.ts](../../packages/opencode/src/tool/tool.ts)
- Tool 注册表: [registry.ts](../../packages/opencode/src/tool/registry.ts)
- Bash tool: [bash.ts](../../packages/opencode/src/tool/bash.ts)
- Task tool: [task.ts](../../packages/opencode/src/tool/task.ts)
- 新权限系统: [next.ts](../../packages/opencode/src/permission/next.ts)
- 旧权限系统: [index.ts](../../packages/opencode/src/permission/index.ts)
- 配置保护: [config-paths.ts](../../packages/opencode/src/kilocode/permission/config-paths.ts)
- 插件 Hook 定义: [index.ts](../../packages/plugin/src/index.ts)
- 插件运行时: [index.ts](../../packages/opencode/src/plugin/index.ts)
- Session tool 注入: [prompt.ts](../../packages/opencode/src/session/prompt.ts)

## Tool 的统一抽象

[tool.ts](../../packages/opencode/src/tool/tool.ts) 定义了所有工具的统一协议：

- `id`
- `description`
- `parameters`
- `execute(args, ctx)`

并且 `Tool.define()` 自动做：

- 参数校验
- 错误格式化
- 输出截断

这意味着企业要做工具治理时，不需要针对每个工具自己发明协议。

## ToolRegistry 是能力入口汇总

[registry.ts](../../packages/opencode/src/tool/registry.ts) 会把三类能力收拢起来：

- 内置工具
- 配置目录下的自定义工具
- plugin 提供的工具

同时它还会根据模型切换：

- `apply_patch`
- `edit / write`

### 这对企业的意义

如果企业想扩能力，优先级可以这样理解：

- 少改核心：加 plugin tool
- 中等改造：加配置目录工具
- 深度改造：改内置工具或 ToolRegistry 策略

## 真正把 tool 暴露给模型的是 `SessionPrompt.resolveTools()`

[prompt.ts](../../packages/opencode/src/session/prompt.ts) 里的 `resolveTools()` 会：

- 从 `ToolRegistry.tools()` 取工具
- 从 `MCP.tools()` 取 MCP 工具
- 包装每个工具的 execute
- 注入 `ctx.ask`
- 触发 plugin hook

### 工具执行前后已有标准 Hook

当前已接好的扩展点包括：

- `tool.definition`
- `tool.execute.before`
- `tool.execute.after`
- `shell.env`

这意味着做这些事已经有现成落点：

- 审计日志
- 参数脱敏
- 工具白名单统计
- 输出附加标签
- shell 环境注入

## `PermissionNext` 是主 agent/tool 链路的权限核心

[next.ts](../../packages/opencode/src/permission/next.ts) 是当前主执行链使用的权限系统。

它负责：

- rule 的 config 化与 ruleset 化
- evaluate
- ask / reply / always
- pending permission 管理
- 配置落盘
- config 路径保护

### 规则来源

主规则通常来自三层：

- agent 自身 permission
- session.permission
- 已保存的全局 permission config

### 规则行为

每条权限规则最终会落成：

- `allow`
- `deny`
- `ask`

默认没有命中时，通常按 `ask` 处理。

## 一个非常重要的现实结论

插件 Hook 里的 `permission.ask` 当前接的是旧的 [permission/index.ts](../../packages/opencode/src/permission/index.ts)，不是主 tool 链路里使用的 `PermissionNext`。

而主 session / tool 执行链中，我们确认实际调用的是：

- `PermissionNext.ask()`

这意味着：

- `permission.ask` 这个插件 Hook 目前不是主工具审批链的可靠扩展点
- 如果企业要对主 agent/tool 链做统一权限治理，应该优先扩展 `PermissionNext`

这是当前仓库里非常值得记录的一个治理边界。

## ConfigProtection 是一个很重要的企业安全细节

[config-paths.ts](../../packages/opencode/src/kilocode/permission/config-paths.ts) 会对以下路径加强保护：

- `.kilo/`
- `.kilocode/`
- `.opencode/`
- 根目录的 `kilo.json`、`opencode.json`、`AGENTS.md`

它的效果是：

- 即使规则算出来是 `allow`
- 如果是配置路径编辑，仍然会强制走 ask
- 同时禁用 “Allow always”

这很适合企业环境，因为配置和策略文件通常不应该被长期豁免。

## `bash` 是高风险工具的典型样板

[bash.ts](../../packages/opencode/src/tool/bash.ts) 展示了一个比较完整的高风险工具治理模式：

- 先解析 shell AST
- 提取命令模式
- 提取可能访问的目录
- 对外部目录单独申请 `external_directory`
- 对命令本身申请 `bash`
- 记录层级规则用于更细粒度授权
- 支持 shell 环境注入
- 实时更新 metadata 输出

这说明系统并不是把 shell 当成一个不可治理黑盒，而是在努力把它结构化。

对企业来说，这是做：

- shell 白名单
- 命令模板审批
- 风险命令识别
- 审计归档

最值得继续下钻的一层。

## `task` 工具体现了子 agent 的越权控制

[task.ts](../../packages/opencode/src/tool/task.ts) 非常值得关注，因为它决定多 agent / 子任务时的权限边界。

当前它做了几件很重要的事：

- 调用前先检查是否允许 `task`
- 为子 session 单独创建权限集
- 继承父调用者的 `edit`、`bash`、MCP 相关限制
- 默认禁用 `todowrite`、`todoread`
- 按情况禁用递归 task

这个设计对企业非常关键，因为它说明系统已经意识到：

- 子 agent 不能天然继承无限能力
- 多层委派必须防止越权扩大

## Plugin 是治理层最稳的扩展面

[packages/plugin/src/index.ts](../../packages/plugin/src/index.ts) 里定义的 Hook，和 [packages/opencode/src/plugin/index.ts](../../packages/opencode/src/plugin/index.ts) 的运行时结合起来，已经提供了多类企业扩展点：

- `chat.params`
- `chat.headers`
- `tool.definition`
- `tool.execute.before`
- `tool.execute.after`
- `shell.env`
- `event`
- `auth`

### 适合放在插件层的企业逻辑

- 工具审计
- header 注入
- 请求归因
- shell 环境注入
- 企业鉴权
- provider options 补充

### 不适合只靠插件层解决的问题

- 主权限引擎改造
- session 调度策略改造
- 工具状态机改造
- 子 agent 权限模型重写

这些更适合直接落在 core 模块里。

## 企业治理的推荐落点

### 场景 1：做工具审计

优先看：

- `tool.execute.before`
- `tool.execute.after`
- Bus 事件
- telemetry

### 场景 2：做工具白名单 / 黑名单

优先看：

- agent.permission
- session.permission
- `PermissionNext.evaluate()`
- `ToolRegistry.tools()`

### 场景 3：做统一审批流

优先看：

- `PermissionNext.ask()`
- `PermissionNext.reply()`
- 对应 UI / route

不要误以为 `permission.ask` plugin hook 就能覆盖主工具链。

### 场景 4：限制子 agent 越权

优先看：

- `task.ts`
- agent permission
- session.permission 继承策略

## 当前最重要的治理结论

- Tool 扩展点很多，但主权限治理点集中在 `PermissionNext`
- Plugin 很适合做审计和协议注入，不适合单独承担主权限引擎
- `bash` 和 `task` 是理解企业级风险控制的两个代表性样板
