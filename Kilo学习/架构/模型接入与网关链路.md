# Provider 与 Gateway 链路

## 为什么这块对企业二开最关键

如果企业要做下面这些事，第一优先级几乎都在 provider 边界：

- 接企业内部模型网关
- 统一管理模型白名单
- 按组织或租户切换模型集
- 自定义鉴权方式
- 控制请求头、路由、超时、缓存、计费
- 做模型能力适配和兼容层

对这个仓库来说，provider 不是“模型枚举”，而是一整套运行时装配系统。

## 关键代码入口

- Provider 主体: [provider.ts](../../packages/opencode/src/provider/provider.ts)
- Provider 鉴权: [auth.ts](../../packages/opencode/src/provider/auth.ts)
- 模型源与动态注入: [models.ts](../../packages/opencode/src/provider/models.ts)
- 动态模型缓存: [model-cache.ts](../../packages/opencode/src/provider/model-cache.ts)
- Provider 适配转换: [transform.ts](../../packages/opencode/src/provider/transform.ts)
- Provider 路由: [provider.ts](../../packages/opencode/src/server/routes/provider.ts)
- Kilo Gateway provider: [provider.ts](../../packages/kilo-gateway/src/provider.ts)
- Kilo Gateway routes: [routes.ts](../../packages/kilo-gateway/src/server/routes.ts)
- Kilo headers: [headers.ts](../../packages/kilo-gateway/src/headers.ts)

## Provider 运行时不是单一来源

`Provider.state()` 会把多种来源合并成最终 provider 运行时。

可以理解成下面这个顺序：

1. `ModelsDev.get()` 提供基础 provider / model 数据
2. `config.provider` 可以扩展或覆盖 provider 和 model
3. 环境变量可以激活 provider
4. `Auth` 中保存的 API key / OAuth 可以激活 provider
5. 插件 auth loader 可以补充 provider options
6. `CUSTOM_LOADERS` 可以为特定 provider 注入自定义加载逻辑
7. 最后再根据 enabled / disabled / whitelist / blacklist 做过滤

这意味着 provider 层已经天然支持多来源合并，而不是死写在代码里。

## Provider 与 Model 的结构

在 [provider.ts](../../packages/opencode/src/provider/provider.ts) 里可以看到两层核心对象：

- `Provider.Info`
  - provider 基本信息
  - 来源 `source`
  - 环境变量列表
  - provider options
  - models

- `Provider.Model`
  - providerID
  - api 信息
  - capabilities
  - cost
  - limit
  - headers
  - variants

这说明企业要做模型治理时，可以同时控制：

- provider 级配置
- model 级配置
- variant 级行为

## `ModelsDev` 是基础模型源，不是唯一模型源

[models.ts](../../packages/opencode/src/provider/models.ts) 里，`ModelsDev.get()` 先读：

- 本地缓存
- 构建期 snapshot
- 远端 `models.dev`

然后再动态注入：

- `kilo`
- `apertis`

### 动态 provider 注入的意义

这点对企业非常重要，因为它说明模型列表可以不是静态文件，而可以是：

- 来自企业网关
- 来自组织配置
- 来自带鉴权的远端 API

`ModelCache` 已经提供了一个现成模式：

- cache-first
- TTL
- in-flight refresh 去重
- 鉴权信息合并
- auth / org 切换后 clear cache

这是企业“组织级模型目录”很好的落点。

## ProviderAuth 通过插件接入认证方式

[auth.ts](../../packages/opencode/src/provider/auth.ts) 说明 provider 认证方式不是写死在 CLI 中，而是通过 plugin 的 `auth` 声明接入。

当前能力包括：

- 列出 provider 的认证方法
- 发起 OAuth authorize
- 处理 OAuth callback
- 写入 API key
- 在成功后更新 telemetry identity
- 清空对应 provider 的 model cache

### 这对企业的意义

如果你的企业内部模型平台需要：

- 设备码登录
- SSO
- API key
- 组织选择

优先考虑用 plugin auth loader 走现有 ProviderAuth 体系，而不是自己另起一套登录入口。

## `getSDK()` 与 `getLanguage()` 才是运行时真正调用模型的地方

在 [provider.ts](../../packages/opencode/src/provider/provider.ts) 中：

- `getSDK()` 负责拿到真实 AI SDK provider 实例
- `getLanguage()` 负责拿到实际的 `languageModel`

这里做了几件关键事：

- 根据 npm 包选择 bundled provider 或动态安装 provider
- 合并 provider options、key、baseURL、headers
- 缓存 SDK 实例和 language model 实例
- 对特定 provider 做 fetch 包装和请求修正

### 企业改造时的建议

如果只是接 OpenAI-compatible 企业网关：

- 优先走 config + provider options + baseURL
- 尽量不要直接修改 `getSDK()`

如果需要 provider 专属行为：

- 优先加 custom loader
- 只有在 SDK 初始化过程必须特殊化时再动 `getSDK()`

## `ProviderTransform` 是 provider 兼容层

[transform.ts](../../packages/opencode/src/provider/transform.ts) 非常重要，它负责：

- 规范化消息格式
- 处理 provider 不兼容的 message / toolCallId / reasoning 格式
- 生成 variants
- 生成 providerOptions
- 处理 schema 兼容性

这一层本质上是在做“统一上层语义，兼容不同下游 provider”。

### 企业最常见的改造点

- 私有网关只接受特定字段
- 某些模型需要特殊 reasoning / thinking 参数
- 某些 provider 需要不同的 schema 形态
- 某些模型不支持附件或多模态

这些问题优先落在 `ProviderTransform`，而不是污染 session 主链。

## Kilo Gateway 的接入方式

Kilo 自己的 provider 在 [packages/kilo-gateway/src/provider.ts](../../packages/kilo-gateway/src/provider.ts)。

它的本质是：

- 以 OpenRouter 风格接口对外
- 自动加 Kilo 特定 headers
- 根据 token 和 organization 解析 baseURL
- 同时暴露 openrouter / anthropic / openai / openaiCompatible 适配面

这说明 Kilo Gateway 本身就是一个“企业网关接入样板”。

### Kilo provider 的几个关键点

- header 统一由 [headers.ts](../../packages/kilo-gateway/src/headers.ts) 构建
- organizationId、taskId、projectId、machineId 都可以入 header
- API baseURL 能随 token 和组织动态变化

这对企业非常有参考价值，因为很多企业平台都会有：

- 组织隔离
- 项目归因
- 机器标识
- 功能归因

## Gateway routes 是 CLI server 的扩展面

[server.ts](../../packages/opencode/src/server/server.ts) 会把 `createKiloRoutes(...)` 注入到 `/kilo` 路由空间。

[routes.ts](../../packages/kilo-gateway/src/server/routes.ts) 里能看到：

- profile
- organization 切换
- organization modes
- notifications
- cloud sessions
- cloud session import
- claw 相关接口

这意味着：

- 企业平台能力不一定非要塞进 session route
- 完全可以按平台域名义挂新 route，再注入 server

这是做企业控制平面能力时非常稳的模式。

## 企业级接入的推荐落点

### 场景 1：接企业内部 OpenAI-compatible 网关

优先路径：

- `config.provider`
- `baseURL`
- `headers`
- `ProviderAuth`
- 必要时少量补 `ProviderTransform`

不要一开始就改 session 或 tool 层。

### 场景 2：按组织动态拉模型目录

优先路径：

- `ModelsDev.get()`
- `ModelCache`
- auth 中的 organization 信息

### 场景 3：做企业统一鉴权

优先路径：

- plugin `auth`
- `ProviderAuth`
- provider options loader

### 场景 4：做 provider 专属协议兼容

优先路径：

- `ProviderTransform`
- custom loader
- 必要时 `getSDK()`

## 这一层的风险点

- 不要把 provider 兼容逻辑写进 session 主链
- 不要把组织级模型目录硬编码成静态表
- 不要把认证逻辑散落到多个 route 里
- 不要忽略 `ModelCache.clear()`，否则组织切换和 token 切换后很容易脏读
