# 策略形成与隔离模型

## 为什么这块对企业级二开很重要

企业真正落地时，通常很快会遇到两个问题：

- 最终执行策略到底是怎么形成的
- 多人、多任务、多分支、多工作区的隔离到底靠什么实现

这两个问题分别主要落在：

- `config + agent + organization modes`
- `project + instance + workspace + worktree`

## 关键代码入口

- 配置主文件: [config.ts](../../packages/opencode/src/config/config.ts)
- agent 定义: [agent.ts](../../packages/opencode/src/agent/agent.ts)
- project 模型: [project.ts](../../packages/opencode/src/project/project.ts)
- instance 作用域: [instance.ts](../../packages/opencode/src/project/instance.ts)
- workspace: [workspace.ts](../../packages/opencode/src/control-plane/workspace.ts)
- workspace 上下文: [workspace-context.ts](../../packages/opencode/src/control-plane/workspace-context.ts)
- worktree: [index.ts](../../packages/opencode/src/worktree/index.ts)

## 配置不是单文件，而是多层合并

[config.ts](../../packages/opencode/src/config/config.ts) 里最值得记住的不是 schema，而是它的合并顺序。

当前大致可以理解成：

1. 旧 Kilocode 遗留配置迁移
2. organization custom modes
3. remote well-known config
4. global config
5. custom config path
6. project config
7. `.kilo/.kilocode/.opencode` 目录配置
8. inline config
9. managed config directory

其中 managed config 是最高优先级。

### 这对企业的意义

这说明仓库已经具备“企业托管配置覆盖用户配置”的基本能力，而不是纯粹的本地用户偏好系统。

## 托管配置目录是企业级控制点

`config.ts` 中的 `managedConfigDir()` 会指向系统级目录：

- macOS: `/Library/Application Support/kilo`
- Windows: `C:\\ProgramData\\kilo`
- Linux: `/etc/kilo`

这层配置高于用户和项目配置。

### 这意味着

企业如果想做：

- 强制 provider 白名单
- 强制 agent 策略
- 强制权限默认值
- 强制插件列表

优先应该利用 managed config，而不是要求开发者每台机器手动改个人配置。

## organization modes 已经进入配置形成链

`config.ts` 会在用户有 `kilo` OAuth 且带 `accountId` 时调用 `fetchOrganizationModes()`，把组织级 modes 转成 agent 注入配置结果。

这说明：

- 组织级执行模式已经不是纯前端概念
- 它会进入 CLI 内核的最终 config
- 后续 agent 决策会实际受它影响

这对企业非常重要，因为它说明“组织策略”已经有现成承载面。

## `agent.ts` 负责把最终 config 变成执行策略

[agent.ts](../../packages/opencode/src/agent/agent.ts) 里可以看到：

- 内建 agent 默认值
- permission 默认值
- 用户 config 覆盖
- organization mode 等外部注入配置覆盖
- `default_agent` 解析

### agent 实际承载的东西

- prompt
- permission
- model
- mode
- options
- steps
- displayName

所以企业如果要做：

- 组织级角色
- 只读模式
- 审批更严格的模式
- 指定模型的专业模式

优先改的不是 session，而是 config 到 agent 的形成链。

## `Project` 决定“当前项目是谁”

[project.ts](../../packages/opencode/src/project/project.ts) 里，`Project.fromDirectory()` 会：

- 根据 `.git` 向上找项目
- 计算 project id
- 找出 worktree 根与 sandbox
- 记录 sandboxes

一个很重要的点是：

- project id 基于 git 根提交推导

这意味着同一仓库的不同 sandbox / worktree 可以被识别为同一 project 生态的一部分。

## `Instance` 负责目录级隔离

`Instance` 是按目录缓存的目录级运行时作用域。

它解决的是：

- 当前请求属于哪个目录
- 对应 project / worktree 是谁
- 哪些状态是该目录独享的

对企业来说，`Instance` 更像“项目级运行容器”。

## `Workspace` 负责逻辑工作区与远端路由

[workspace.ts](../../packages/opencode/src/control-plane/workspace.ts) 表明 workspace 不是简单本地目录别名，而是控制平面实体。

它支持：

- create
- list
- get
- remove
- startSyncing

并且会通过 adaptor 与远端交互，拉取 `/event` SSE 流。

### 这意味着

如果企业要做：

- 远端开发环境
- 云工作区
- 控制平面统一编排

workspace 是比 worktree 更高一级的抽象。

## `Worktree` 是本地 git 隔离样板

[worktree/index.ts](../../packages/opencode/src/worktree/index.ts) 展示了本地隔离的完整样板：

- 自动生成名称与分支
- `git worktree add`
- bootstrap 新目录实例
- 运行项目启动脚本
- remove
- reset

同时它还会：

- 把新目录加入 project sandboxes
- 发出 ready / failed 事件

### 这对企业很有价值

如果企业要做：

- 多任务并行开发
- 分支级隔离
- 实验环境隔离
- 任务沙箱

本地侧可以直接沿 worktree 这条链扩展。

## workspace 与 worktree 的关系

可以先这样理解：

- `worktree` 偏本地 git 隔离
- `workspace` 偏控制平面与远端抽象

两者不是一回事。

企业级能力往往会同时用到：

- 本地并行开发用 worktree
- 远端 / 云环境用 workspace

## 对企业接管的直接启发

### 场景 1：做组织级强管控

优先看：

- managed config
- organization modes
- agent 合并逻辑

### 场景 2：做团队并行开发隔离

优先看：

- project sandboxes
- worktree create / reset / remove

### 场景 3：做远端工作区或云开发

优先看：

- workspace
- adaptor
- workspace router middleware

### 场景 4：做统一策略平台

优先看：

- `config -> agent` 的形成链
- 不要直接把组织规则硬编码进 session 流程

## 当前结论

企业级二开里，“策略”与“隔离”两件事不应该混着做：

- 策略主要走 config / agent / organization modes
- 隔离主要走 project / instance / workspace / worktree

把这两个面分清楚，后续设计会稳很多。
