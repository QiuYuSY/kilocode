# 2026-04-07 企业扩展边界笔记

## 本轮主题

围绕企业级二次开发，拆清三条关键边界：

- provider / gateway
- tool / permission / plugin
- telemetry / route 注入

## 最重要的新结论

### 1. provider 是运行时装配系统，不是模型列表

它会融合：

- models.dev
- config
- env
- auth
- plugin auth loader
- custom loader
- dynamic model cache

所以企业接私有模型平台时，优先应该沿 provider 边界扩展，而不是从 session 或前端切入。

### 2. Kilo Gateway 本身就是一个企业网关接入样板

它已经展示了：

- 动态模型拉取
- organization 维度隔离
- header 归因
- profile / modes / notifications / cloud sessions route 注入

后面如果我们接企业内部平台，可以把它当参考实现。

### 3. 主工具权限治理核心是 `PermissionNext`

这轮最值得记住的一点是：

- 主 agent / tool 链用的是 `PermissionNext.ask`
- plugin 里的 `permission.ask` 接的是旧 `Permission`

这意味着企业如果要做真正可靠的审批流和权限策略，不应该误把 plugin hook 当成主治理入口。

### 4. plugin 层更适合做审计和协议注入

当前稳定可用的企业扩展点包括：

- `chat.params`
- `chat.headers`
- `tool.definition`
- `tool.execute.before`
- `tool.execute.after`
- `shell.env`
- `auth`

这层很适合做：

- 审计
- 归因
- 环境注入
- 认证适配

### 5. `bash` 和 `task` 是企业风险控制的两个样板

- `bash` 展示了命令与目录的结构化审批
- `task` 展示了子 agent 权限继承和越权防护

如果未来要做企业级策略，这两个文件会很值得反复看。

## 对最终目标的影响

这轮之后，企业级二开的主战场已经比较明确：

- 模型平台接入：provider
- 权限与审批：PermissionNext + tool
- 审计与归因：plugin + telemetry
- 组织能力：gateway route + auth + model cache

## 下一轮建议

继续围绕企业接管往下走的话，最值得拆的是：

- `agent + config` 如何形成组织级策略体系
- `workspace + worktree` 如何支撑多人并行和隔离
- `server route + sdk` 如何扩展企业后端能力
