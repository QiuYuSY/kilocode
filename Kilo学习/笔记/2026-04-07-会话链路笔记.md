# 2026-04-07 Session 链路笔记

## 本轮主题

用户发消息后，Kilo 的 session 是怎样跑起来的。

## 新确认的结构

### 1. `SessionPrompt` 是调度中枢

之前知道 session route 很重要，但这轮确认后可以更准确地说：

- route 只负责 HTTP 协议层
- `SessionPrompt` 才是对话调度中枢

它控制：

- 创建用户消息
- session busy / cancel / resume
- 选择 compaction、subtask 或 normal processing
- 组装 prompt 和 tools
- 驱动 processor

### 2. `SessionProcessor` 是执行状态机

它不负责决定“这轮要不要跑”，而是负责：

- 消费模型流
- 更新 text / reasoning / tool parts
- 写 token / cost / finish
- 处理 retry、patch、summary、abort

这个分层很重要，因为以后如果我们改“调度策略”和“执行策略”，应该分别落到不同模块里。

### 3. 工具系统在 prompt 阶段接入，不在 processor 阶段定义

`resolveTools()` 发生在 `SessionPrompt.loop()` 中，而不是在 `SessionProcessor` 内部。

这意味着：

- 工具可用性是调度阶段决定的
- 真正的工具调用结果处理是在 processor 阶段完成的

### 4. Agent 其实是一组执行策略，不只是一个标签

`agent.ts` 里的 agent 同时决定：

- prompt
- permission
- model
- steps
- mode

这意味着 agent 是企业二开里非常重要的策略封装层。

## 当前最关键的认识

如果把一次 session 看成一个小型工作流，那么：

- `route` 是入口
- `prompt` 是调度器
- `processor` 是执行器
- `llm` 是模型适配层
- `tool registry + MCP` 是能力供应层
- `agent` 是策略配置层

## 下一轮建议

继续拆两条边界：

- `provider/*`: 模型、网关、鉴权、header、成本
- `tool/*`: 工具定义、权限、插件化、审计与治理
