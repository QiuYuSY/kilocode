# 2026-04-07 启动链路与作用域笔记

## 本轮主题

第一条主线：`kilo serve` 如何启动，HTTP 请求如何落到正确的目录与 workspace。

## 新确认的关键事实

### 1. `serve` 只是入口，真正的关键是请求作用域绑定

仅仅知道 `ServeCommand` 会启动 `Server.listen()` 还不够。更重要的是，服务端会在请求进入时显式建立：

- `WorkspaceContext`
- `Instance` 上下文

这两层作用域决定了后续模块看到的“当前项目”和“当前工作区”是谁。

### 2. instance 是按目录缓存的，不是按请求创建的

`Instance.provide()` 会按 `directory` 复用上下文，这意味着：

- 同一个目录的请求共享实例状态
- state、watcher、LSP 等能力不会每次请求都重建
- 企业扩展如果想做项目级缓存，顺着这个模型做最自然

### 3. bootstrap 初始化的是“项目级能力包”

`InstanceBootstrap()` 初始化的不是某个单点能力，而是一组目录级运行时依赖，例如：

- LSP
- FileWatcher
- VCS
- Snapshot
- Plugin

这说明项目实例更像一个“按目录启动的小型运行容器”。

### 4. workspace 设计已经在为远端化做准备

`WorkspaceRouterMiddleware` 表明请求未来或现在都可能被转发到别处处理，而不是永远在本地 server 内完成。

## 对二开的意义

- 做项目级策略、缓存、限制时，优先看 `Instance`
- 做远端路由、多环境、控制平面时，优先看 `Workspace`
- 不要把目录级能力误做成粗暴的进程级全局状态
