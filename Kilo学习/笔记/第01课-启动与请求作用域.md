# 第01课：启动与请求作用域

## 本课目标

搞清楚两个问题：

1. `kilo serve` 是怎么启动起来的
2. 一个请求进入后，为什么系统能知道它属于哪个 workspace、directory、project、worktree

## 为什么从这里开始

这是整个内核的总入口。

如果这一层没吃透，后面看：

- session
- tool
- permission
- provider
- worktree

都会变成零散知识点，串不起来。

## 本课核心结论

### 1. `serve` 是入口，不是核心逻辑本体

`serve` 负责：

- 解析网络参数
- 调用 `Server.listen(opts)`
- 处理优雅退出

但真正重要的是 `Server.listen()` 之后，request middleware 如何建立请求作用域。

### 2. 请求作用域有两层

第一层：`WorkspaceContext`

- 表示当前请求属于哪个逻辑 workspace
- 主要影响远端路由与控制平面能力

第二层：`Instance`

- 表示当前请求绑定到哪个本地目录实例
- 提供 `directory`、`worktree`、`project`
- 决定项目级状态如何缓存和复用

### 3. `Instance` 是按目录缓存的项目级运行容器

这意味着：

- 同目录请求会复用状态
- 不同目录天然隔离
- 企业二开里很多项目级扩展应该挂在 instance state 上

### 4. `InstanceBootstrap` 初始化的是项目级能力包

例如：

- Plugin
- LSP
- FileWatcher
- File
- VCS
- Snapshot

所以它不是一个普通初始化函数，而是在建立“该目录对应的运行时环境”。

### 5. workspace 和 worktree 不是一回事

这一课先建立初步认知：

- workspace 偏控制平面、远端工作区
- worktree 偏本地 git 隔离
