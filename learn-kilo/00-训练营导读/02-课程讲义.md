# 00 课程讲义

## 为什么旧式源码学习很容易失效

多数“带你看项目”的资料会停在 3 个层面:

- 告诉你有哪些目录
- 告诉你有哪些模块
- 告诉你可以从哪里开始看

这些信息不是没用，但它们不够支撑二次开发。

二次开发真正需要的是另外 4 种能力:

### 1. 结构判断力

面对一个需求时，你要先判断它属于:

- 核心运行时
- 客户端接入
- SDK 契约
- Kilo 平台差异

### 2. 设计解释力

你要能解释:

- 为什么这里需要 `Instance`
- 为什么 `Session` 不是聊天记录
- 为什么工具不是静态列表
- 为什么扩展端必须共享后端连接

### 3. 改动落点能力

你要能回答:

- 这个功能先改哪一层
- 会影响哪些包
- 是否需要 regen SDK
- 是否属于 Kilo 专属差异

### 4. 交付能力

你要知道:

- 如何跑测试
- 如何验证 CLI backend 和扩展端
- 如何保持和 upstream 的 merge 友好
- 什么地方该加 `kilocode_change`

## 这套课的新方法

新版课程采用的是“证据链学习法”。

每学一个结论，你都要能给出证据链:

- 它落在哪个包
- 它对应哪个文件
- 它依赖哪个抽象
- 它在运行链上处于什么位置

## 什么叫“达到能做二开”

这里的标准不是让你立刻做大功能，而是达到下面这条线:

- 能完成一个小型功能改动
- 能控制改动边界
- 能说清为什么这么改
- 能做基础验证

## 这套课后面的难点

后面的难点不是代码量，而是你要不断回答“为什么”:

- 为什么核心在 `packages/opencode`
- 为什么 `serve` 是系统中枢
- 为什么 `server.ts` 不只是 route 聚合
- 为什么 `SessionProcessor` 不是多余的一层
- 为什么 `ToolRegistry` 不是写死数组
- 为什么 Kilo 的 fork 差异要尽量集中

## 你应该怎么学

最推荐的姿势只有一句话:

把自己当成未来要在这个仓库里交付改动的人，而不是旁观者。
