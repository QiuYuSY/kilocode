# 01 课程讲义

## 一定先纠正的系统观

KiloCode 的主体不在 `packages/kilo-vscode/`，而在 `packages/opencode/`。

这是理解整个仓库的第一道坎。

因为:

- CLI 入口在这里
- `kilo serve` 在这里
- Session、Agent、Tool、Permission 都在这里
- 其它客户端都是通过 HTTP + SSE 接入这个核心

## 先按层看，不要先按文件看

### 第一层: 核心运行时

`packages/opencode/`

这是最重要的一层。你后面所有“为什么”基本都要回到这里。

### 第二层: 客户端接入

- `packages/kilo-vscode/`
- `packages/app/`
- `packages/desktop/`

这一层不是重复实现核心逻辑，而是把核心运行时接入不同宿主。

### 第三层: 契约与共享

- `packages/sdk/js/`
- `packages/kilo-ui/`
- `packages/plugin/`

这层解决的是一致性和复用，不是业务主干本身。

### 第四层: Kilo 平台差异

- `packages/kilo-gateway/`
- `packages/kilo-telemetry/`
- `packages/opencode/src/kilocode/`

这层体现的是 fork 之后的真实平台能力，而不是表面品牌差异。

## 仓库里你最先要掌握的 3 类命令

### 运行类

- 根目录开发: `bun run dev`
- 扩展开发: `bun run extension`
- CLI 直接跑: `bun run --cwd packages/opencode --conditions=browser src/index.ts`

### 验证类

- 类型检查: `bun turbo typecheck`
- CLI 测试: 在 `packages/opencode/` 下运行 `bun test`
- 单测: 在 `packages/opencode/` 下运行 `bun test test/tool/tool.test.ts`

### 交付类

- SDK regen: 根目录运行 `./script/generate.ts`
- source links: 根目录运行 `bun run script/extract-source-links.ts`
- `kilocode_change` 检查: 在 `packages/kilo-vscode/` 下运行 `bun run check-kilocode-change`

## 开发环境相关的 4 条硬约束

### 1. 这是 monorepo

所以你不能只盯着一个包看功能。

### 2. 这是 fork

所以你必须时刻考虑和 upstream 的 merge 友好。

### 3. Kilo 专属改动优先放进 Kilo 专属路径

例如:

- `packages/opencode/src/kilocode/`
- `packages/kilo-gateway/`

### 4. 改 server endpoint 要考虑 SDK

这不是“后端自己改完就结束”的项目。

## 这一课真正重要的结论

这一课最大的收获应该是:

以后你看到一个需求，先判断它属于哪一层。

如果这一步不会，后面看再多实现也会继续乱。

## 脱离源码版主讲

如果你现在不打开任何代码页，只需要先记住这一课的 3 个判断:

### 判断 1

KiloCode 的主体是核心运行时，不是扩展端。

### 判断 2

仓库不是一坨代码，而是四层结构:

- 核心运行时
- 客户端接入
- 契约与共享
- Kilo 平台差异

### 判断 3

以后接需求时，第一步不是找文件，而是先判断需求属于哪一层。

只要这 3 个判断稳定了，后面你看代码才不会再被目录和文件名带着走。

## 关键抽象解剖卡

### 抽象卡 1: Monorepo 分层

它是什么:

一个把不同职责隔离开的工程结构。

为什么必须存在:

因为 KiloCode 同时有核心运行时、多个客户端、共享契约和 Kilo 平台差异。

没有它会怎样:

所有能力搅在一起，后续二开很难判断边界。

### 抽象卡 2: 核心运行时层

它是什么:

系统真正执行 AI 编程任务的主体。

为什么必须存在:

因为所有客户端都要连接同一套执行系统。

没有它会怎样:

每个客户端都要各做一套后端，系统会碎裂。

### 抽象卡 3: Kilo 差异层

它是什么:

fork 之后 Kilo 专属的平台能力集合。

为什么必须存在:

因为 Kilo 不只是 OpenCode 的 UI 变体，而是有自己的 auth、gateway、telemetry、migration。

没有它会怎样:

Kilo 专属代码会散落在共享文件里，后续 merge upstream 成本极高。

## 常见二开场景

### 场景 1: “我想给 Agent 增一个工具”

先判断:

这是核心运行时层需求。

### 场景 2: “我想给 VS Code 侧栏加个入口”

先判断:

这是客户端接入层需求。

### 场景 3: “我加了新 route，扩展也要调”

先判断:

这同时涉及核心运行时层和契约层。

### 场景 4: “我想加 Kilo 专属的组织配置能力”

先判断:

这属于 Kilo 平台差异层。
