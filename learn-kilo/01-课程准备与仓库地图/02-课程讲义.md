# 讲义

## 一、先纠正一个常见误解

很多人第一次打开这个仓库，会下意识做一件事：

- 看见 VS Code 扩展目录最显眼
- 就以为这仓库主要是个插件项目

这是第一层误解。

当前仓库更准确的理解是：

它是一个以 CLI 运行时为核心的多产品 monorepo。

什么意思？

意思是这里不是“一个插件，顺便带一些脚本”，而是：

- 有一个核心引擎
- 有多个客户端
- 有共享桥梁层
- 有 Kilo 特有的增量能力

如果你第一步不把这个认知立住，后面无论看哪个目录，都会很容易误判它的重要性。

## 二、先用最粗的方式分层

第一次理解这类仓库，不要先按文件分，先按“包的角色”分。

从当前仓库实际结构来看，最适合小白的第一层分法是 4 类。

### 1. 核心引擎层

最重要的是：

- [packages/opencode](/d:/Code/kilocode/packages/opencode)

这个包不是普通工具包，它是整个系统的心脏。  
你之后会在这里看到：

- CLI 入口
- 命令系统
- Hono 服务端
- Session 数据模型
- Agent 模式
- Tool 注册与编排
- Provider
- Permission / Question
- Storage / DB

所以你应该先形成一个非常稳的判断：

这个仓库里最重要的地方，不是扩展，不是 UI，而是 `packages/opencode`。

### 2. 客户端层

核心客户端主要是：

- [packages/kilo-vscode](/d:/Code/kilocode/packages/kilo-vscode)
- [packages/app](/d:/Code/kilocode/packages/app)
- [packages/desktop](/d:/Code/kilocode/packages/desktop)

它们的关系不是“各自实现一套 AI 功能”，而是：

- `kilo-vscode` 负责 VS Code 场景
- `app` 负责共享前端应用层
- `desktop` 负责桌面壳

它们都不是“另一个核心引擎”，而是不同形态的客户端。

### 3. 共享桥梁层

这里最该优先认识的是：

- [packages/sdk/js](/d:/Code/kilocode/packages/sdk/js)
- [packages/kilo-ui](/d:/Code/kilocode/packages/kilo-ui)
- [packages/plugin](/d:/Code/kilocode/packages/plugin)
- [packages/util](/d:/Code/kilocode/packages/util)

它们的共同特点是：

- 自己不是最终产品
- 但多个产品都依赖它们

这里尤其要强调 [packages/sdk/js](/d:/Code/kilocode/packages/sdk/js)。

很多新手会把 SDK 看成“以后再看也行”的层。其实不对。  
因为它承担了：

- 服务端接口契约
- 多客户端共享调用方式
- 客户端和后端之间的稳定桥梁

### 4. Kilo 增量层

当前仓库不是纯 upstream OpenCode，而是一个 fork 之后继续发展的系统。

这意味着有一层非常重要的 Kilo-specific 能力，主要集中在：

- [packages/kilo-gateway](/d:/Code/kilocode/packages/kilo-gateway)
- [packages/kilo-telemetry](/d:/Code/kilocode/packages/kilo-telemetry)
- [packages/kilo-i18n](/d:/Code/kilocode/packages/kilo-i18n)
- [packages/opencode/src/kilocode](/d:/Code/kilocode/packages/opencode/src/kilocode)

这层的作用不是替代核心引擎，而是在 upstream 核心之上增加 Kilo 平台自己的能力。

## 三、为什么 `packages/opencode` 必须先学

这一点你必须真正理解，而不是机械记住。

### 原因 1：它控制主链路

从启动到执行，最关键的路径基本都在这里：

- 入口
- 命令
- 服务端
- Session
- Agent
- Tool

如果这条主链路你没吃透，后面看扩展端只会看成很多消息转来转去的代码。

### 原因 2：其他产品都绕着它转

VS Code 扩展不是自己实现一套 Agent runtime。  
Web / Desktop 也不是自己维护一套 Session 执行系统。

它们都在不同程度上依赖核心运行时。

### 原因 3：架构理解必须先从中枢开始

如果你先看客户端，很容易只看到“界面怎么渲染”“命令怎么发出去”，却不知道这些东西最后流到了哪里。

## 四、为什么 `packages/kilo-vscode` 不能单独理解

很多人会在这里掉坑。

看起来扩展目录很完整：

- 有入口
- 有服务
- 有 webview
- 有 Agent Manager

但如果你把它当成一个独立系统去学，会立刻遇到问题：

- 你不知道它为什么要启动 `kilo serve`
- 你不知道为什么会有 SDK client
- 你不知道 SSE 事件从哪里来
- 你不知道 Session 数据为什么长成那样

原因很简单：

扩展是核心引擎的客户端，不是核心引擎本身。

所以你后面学扩展课之前，必须先把核心引擎的主链路理解清楚。

## 五、为什么 `packages/sdk/js` 这么关键

第一课里，你不需要学会 SDK 代码细节。  
但你必须先建立正确地位判断：

它不是辅助工具，而是桥梁。

在这个仓库里，SDK 的核心价值是：

- 把服务端能力稳定地暴露给客户端
- 让多个客户端不用各自手写一套调用方式
- 成为“后端 API 变化 -> 客户端调用变化”的中间契约层

这也是为什么后面课程里会单独讲 SDK，而不是把它顺带提一下。

## 六、为什么后面课程要按“核心 -> 客户端 -> Kilo 差异”来排

你现在可以理解这套课程顺序不是随便拍脑袋定的。

### 先学核心

因为核心引擎决定了系统真正的主链路。

### 再学客户端

因为客户端本质上是在承接核心能力，而不是独立产生核心能力。

### 最后学 Kilo 差异

因为只有在你已经知道“通用核心长什么样”之后，才知道 Kilo 额外加了什么。

否则你会把 upstream 核心和 Kilo 增量混成一团。

## 七、这节课最重要的 5 个结论

请你最后一定要带走这 5 个判断：

1. 这是一个多产品 monorepo，不是单插件项目。
2. [packages/opencode](/d:/Code/kilocode/packages/opencode) 是核心引擎，必须第一优先级学习。
3. [packages/kilo-vscode](/d:/Code/kilocode/packages/kilo-vscode) 是重要客户端，但不是系统主体。
4. [packages/sdk/js](/d:/Code/kilocode/packages/sdk/js) 是客户端与后端之间的共享桥梁。
5. 后续学习顺序必须是“先核心，再客户端，再 Kilo 差异”。

如果你能真正说清这 5 点，第一课才算真的起作用。
