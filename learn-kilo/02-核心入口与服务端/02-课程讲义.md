# 讲义

## 一、从“程序开始执行”到“系统开始成型”

很多初学者第一次打开 [packages/opencode/src/index.ts](/d:/Code/kilocode/packages/opencode/src/index.ts) 时，会自然地把它理解成：

- 一个程序入口
- 一个命令注册文件

这两个判断并不完全错，但都太浅了。

更准确的说法应该是：

这个入口文件在做“系统级初始化编排”。

为什么要这么理解？

因为你在这一个文件里，能同时看到下面几类事情：

- 命令注册
- 日志初始化
- 环境变量注入
- 遥测初始化
- 一次性迁移
- 异常处理
- 退出清理

如果你把它只看成命令入口，就会忽略一个很重要的工程事实：

对于像 Kilo 这样的系统，入口文件不只是告诉程序“从哪开始”，它还在告诉系统“之后所有行为遵守什么启动约定”。

## 二、为什么这节课要从 `index.ts` 而不是从某个产品入口开始

你可能会问：

- 为什么不先看 VS Code 扩展入口？
- 为什么不先看某个具体命令？

原因是：

如果你先看客户端入口，你容易只看到“消息是怎么发出去的”；  
如果你先看某个具体命令，你容易只看到“某个局部能力怎么写”。

但当前系统最重要的理解顺序应该是：

1. 核心引擎怎样启动
2. 核心引擎怎样变成服务
3. 客户端怎样接它

所以本课必须先看 `packages/opencode/src/index.ts`。

## 三、入口文件的第一层：命令注册

你在 `index.ts` 中最容易首先注意到的是 `.command(...)`。

这是正常的，因为这确实是这份文件最显眼的一层：

- `RunCommand`
- `ServeCommand`
- `SessionCommand`
- `McpCommand`
- `AgentCommand`
- `ModelsCommand`

从学习角度，你第一次不需要记住所有命令。

你只要先把它们粗分成两类就够了：

### 1. 直接面向用户的交互命令

例如：

- `run`
- `session`
- `auth`
- `models`

### 2. 面向系统能力暴露的命令

最重要的就是：

- `serve`

这一步很重要，因为它会帮助你理解：

这个仓库不是“一个只能在终端里用的 CLI 工具”，而是“一个核心运行时，可以既作为 CLI 使用，也作为后端服务运行”。

## 四、入口文件的第二层：全局初始化

这一层才是很多人第一次阅读最容易漏掉的地方。

你会看到入口里在做这些事：

### 1. 日志系统初始化

程序一开始就决定后续日志怎么打、级别是什么、是否打印到 stderr。

### 2. 环境变量注入

例如这里会给进程补一些运行时标识，帮助后续模块知道自己当前所处的执行环境。

### 3. 遥测初始化

这不是“附加功能”，而是整个系统观测能力的起点。

### 4. legacy 迁移

这说明系统并不是一次性静态写完的，它有历史、有兼容、有真实产品演进过程。

### 5. 数据迁移

你能看到数据库初始化与迁移逻辑在入口阶段发生。  
这意味着系统启动时已经在考虑数据层的正确性，而不是等出了错再说。

## 五、为什么 finally 区域值得认真看

很多人读入口文件时，会认真看上半部分，却草草略过 `finally`。

这是一个常见错误。

在 [packages/opencode/src/index.ts](/d:/Code/kilocode/packages/opencode/src/index.ts) 里，`finally` 的价值非常大，因为它告诉你：

- 这个系统不是一次性脚本
- 它非常在意清理进程资源
- 它不能指望所有子进程自己优雅退出

这里你能看到的几个关键动作：

- 遥测关闭
- `Instance.disposeAll()`
- 显式 `process.exit()`

为什么这很重要？

因为这说明它服务的是长生命周期、带子进程、带上下文的真实运行时，而不是只跑完就结束的脚本。

## 六、为什么这一课要重点看 `serve`

接下来要看 [packages/opencode/src/cli/cmd/serve.ts](/d:/Code/kilocode/packages/opencode/src/cli/cmd/serve.ts)。

从产品理解角度，`serve` 这个命令非常关键，因为它揭示了整个系统的架构设计：

核心引擎不是只能直接在终端里运行，它还能把自己暴露成一个服务。

这意味着：

- VS Code 扩展可以连接它
- Web / Desktop 可以共享同一类能力
- 后续的 SDK、SSE、客户端通信才有成立基础

### `serve` 做的关键事情

第一眼你应该先只记住下面 5 件事：

1. 解析网络参数
2. 调用 `Server.listen(...)`
3. 打印监听地址
4. 在本地开发模式下做额外同步
5. 处理优雅关闭

其中最值得认真理解的是：

- 它为什么必须处理关闭

因为一旦系统能以服务进程形式存在，就必须认真处理：

- SIGTERM
- SIGINT
- SIGHUP

否则客户端断开、终端关闭、扩展重启时，都会留下僵尸状态。

## 七、为什么 `server.ts` 不是普通“接口文件”

现在看 [packages/opencode/src/server/server.ts](/d:/Code/kilocode/packages/opencode/src/server/server.ts)。

这一份文件特别容易被误读成：

- “一个很长的 route 列表”

但如果你这样看，就会把它的真正意义看浅。

它其实同时承担 4 层职责：

### 1. 安全边界

这里集中处理：

- auth
- CORS
- 错误处理

### 2. 上下文边界

它把请求里的：

- `directory`
- `workspace`

转换成后续模块都能使用的运行时上下文。

### 3. 能力边界

它把系统能力按路由族切开：

- project
- session
- provider
- permission
- question
- telemetry

### 4. 实时同步边界

它通过 `/event` 暴露整个系统的 SSE 事件流。

## 八、为什么 `directory` 注入如此重要

你在 `server.ts` 中会看到一个非常关键的中间段：

- 从 query 或 header 里拿 `directory`
- 做路径解析
- 通过 `Instance.provide(...)` 建立上下文

这一步特别重要。

为什么？

因为后面的很多模块都默认假设：

- “当前我是在某个具体项目上下文里运行的”

而这个“当前上下文”不是凭空来的，就是在服务端这一层建立的。

如果没有这层注入：

- 项目识别会漂
- Session 可能归错目录
- 配置读取可能拿错作用域

所以 `server.ts` 不只是接口入口，它还是上下文建立点。

## 九、为什么 SSE 对这个项目不是可选项

这里一定要打破一个误区：

- “SSE 只是为了看起来更实时一点”

不对。

在这个系统里，执行过程本身就是流式的。

会不断产生的状态包括：

- 文本输出
- tool 调用
- tool 结果
- permission 请求
- question 请求
- session 状态变化

如果没有 SSE，你只剩两种很差的选择：

### 1. 不断轮询

问题：

- 浪费资源
- 延迟高
- 客户端逻辑更乱

### 2. 只看最终结果

问题：

- 丢失过程
- 无法展示细粒度执行
- 无法及时处理 permission / question

所以 `/event` 在这里不是锦上添花，而是让“执行过程可观察、可同步”的关键机制。

## 十、你这一课真正要带走的结论

请不要只记住“看了哪些文件”。

你真正要记住的是：

1. 入口文件在统一初始化整个系统。
2. `serve` 让核心引擎具备了后端服务形态。
3. `server.ts` 同时在建立安全边界、上下文边界、能力边界和实时同步边界。
4. SSE 之所以必要，不是因为酷，而是因为系统执行过程本身就是流式的。

如果这四点你能真正讲出来，这节课才算学到了东西。
