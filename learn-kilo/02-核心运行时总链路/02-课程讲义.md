# 02 课程讲义

## 第一步: 看 `index.ts` 时不要被命令列表淹没

`packages/opencode/src/index.ts` 表面上在做命令注册，实际上它还在做全局初始化。

你要重点看 4 件事:

- 日志初始化
- 环境变量注入
- Telemetry 初始化
- 一次性数据库迁移

这说明入口文件不是“命令清单”，而是“进程启动编排点”。

## 第二步: `serve.ts` 为什么是系统中枢入口

`packages/opencode/src/cli/cmd/serve.ts` 做的不是简单转发。

它把核心运行时暴露成服务端形态，同时处理:

- 网络参数解析
- server 启动
- 本地开发时的 workspace sync
- 进程级优雅关闭
- `Instance.disposeAll()` 的清理

这意味着:

`serve` 不是一个附属命令，而是所有客户端最终依赖的统一后端入口。

## 第三步: `server.ts` 真正建立了 4 个边界

### 1. 安全边界

通过 basic auth 和 CORS 决定谁能访问。

### 2. 请求边界

通过 middleware 注入:

- `workspace`
- `directory`
- `Instance` 上下文

### 3. 能力边界

通过 route 注册暴露:

- project
- session
- permission
- provider
- Kilo routes

### 4. 实时同步边界

通过 SSE / websocket 把运行中的事件流推给客户端。

## 第四步: 为什么 `directory` 这么重要

在 `server.ts` 中，请求会解析出 `directory`，然后进入:

`Instance.provide({ directory, init: InstanceBootstrap, fn })`

这意味着系统不是在全局单例里运行，而是在“按目录组织的运行时上下文”里运行。

这也是为什么:

- 不同 worktree 可以隔离
- 配置和状态能按项目分桶
- 后续 Session / Permission / Config 才有清晰边界

## 第五步: 为什么 SSE 不是附属功能

KiloCode 的执行过程天然是流式的:

- 文本在流
- reasoning 在流
- tool-call / tool-result 在流
- permission / question 在流

如果没有实时事件流，客户端几乎不可能优雅呈现真实执行过程。

所以:

SSE 不是“锦上添花”，而是系统可观察性的基础设施。

## 本课真正要吃透的设计判断

- 入口不是命令清单，而是进程装配点
- `serve` 不是普通命令，而是核心服务化入口
- `server.ts` 不是 route 大杂烩，而是系统边界建立点
- `directory` 不是参数细节，而是运行时上下文的锚点

## 脱离源码版主讲

如果你不看代码，也应该先能完整讲出这节课:

KiloCode 的核心运行时先通过 CLI 入口完成全局装配，再通过 `serve` 暴露成统一后端，随后在服务端里按请求建立上下文和事件流，最后让所有客户端都接入这同一个运行时世界。

这节课最重要的不是某一行实现，而是下面 4 个工程判断:

1. 入口文件负责装配，不只是注册命令
2. `serve` 负责把系统统一服务化
3. `server.ts` 负责建立边界
4. `directory` 决定运行时上下文落在哪个实例里

## 关键抽象解剖卡

### 抽象卡 1: CLI 入口

它是什么:

整个进程的启动装配点。

为什么必须存在:

因为日志、环境、迁移、Telemetry 都要在这里统一初始化。

没有它会怎样:

系统启动逻辑会分散到各处，后续客户端和命令行为难以保持一致。

### 抽象卡 2: `serve`

它是什么:

把核心运行时统一暴露成 headless server 的入口。

为什么必须存在:

因为多个客户端都要依赖同一个后端世界。

没有它会怎样:

客户端必须各自带后端，系统会重复实现。

### 抽象卡 3: 服务端上下文注入

它是什么:

根据请求把 `directory`、`workspace` 和 `Instance` 绑定起来的过程。

为什么必须存在:

因为 KiloCode 不是全局单例运行，而是按项目上下文运行。

没有它会怎样:

状态会串，worktree 和配置边界会混乱。

## 常见二开场景

### 场景 1: 新增一个服务端 route

你应该先想到:

- 它挂在哪个 route 分组
- 它是否依赖 `Instance` 上下文
- 它是否要进入 SDK

### 场景 2: 修改服务启动行为

你应该先想到:

- 这是入口装配问题还是 `serve` 生命周期问题
- 会不会影响 CLI backend 客户端

### 场景 3: 处理服务端请求上下文错误

你应该先想到:

- `directory` 是否正确传入
- `Instance.provide(...)` 是否正确建立
