# 03 课程讲义

## 一、为什么 Session 不是聊天记录

看 `packages/opencode/src/session/index.ts` 时，你会发现 Session 关联的不只是消息。

它还承载:

- `projectID`
- `directory`
- `permission`
- `summary`
- `revert`
- `diff`
- 时间状态

这说明 Session 的本质不是“聊天存档”，而是“任务执行容器”。

## 二、为什么 Message 要拆成 Part

看 `packages/opencode/src/session/message-v2.ts`，你会发现消息不是一坨文本。

系统把过程拆成了很多 Part:

- `text`
- `reasoning`
- `tool`
- `step-start`
- `step-finish`
- `patch`
- `file`
- `agent`

这样做的意义是:

系统可以把 AI 执行过程作为可观察的结构化事件来保存，而不是只在最后留下一段回答。

## 三、`SessionProcessor` 的价值是什么

`packages/opencode/src/session/processor.ts` 真正负责把 LLM 和工具的流式结果收束进 Session 世界。

它处理的不是单一文本输出，而是完整执行循环:

- reasoning-start / delta / end
- tool-input-start
- tool-call
- tool-result / tool-error
- step-start / finish-step

所以它的角色更像“执行事件收束器”。

没有它:

- 会话里的状态很难持续更新
- 工具调用无法优雅落库
- UI 很难完整重建执行过程

## 四、为什么 Agent 不只是 prompt

`packages/opencode/src/agent/agent.ts` 里最重要的不是 prompt 文本，而是 Agent 的行为边界。

Agent 定义了:

- mode
- permission
- model
- native / hidden / deprecated
- steps

也就是说:

一个 Agent 到底能做什么，是由 prompt + permission + 运行模式一起决定的。

## 五、Permission 为什么属于运行时

看 `packages/opencode/src/permission/next.ts` 时，要特别注意:

- 规则如何从 config 转成 ruleset
- 运行时如何 `evaluate(...)`
- `ask(...)` 怎样把请求挂起
- `reply(...)` 怎样继续执行

这说明 permission 不是 UI 弹窗，而是系统执行路径的一部分。

## 六、ToolRegistry 为什么是动态编排层

`packages/opencode/src/tool/registry.ts` 并不是写一个工具数组就结束。

它还要考虑:

- 当前 model 是什么
- 是否使用 `apply_patch`
- 当前 config 是否启用实验工具
- 是否加载 plugin/custom tool
- 当前 Agent 权限边界如何

所以 ToolRegistry 不是“有哪些工具”，而是“当前上下文下应暴露哪组工具”。

## 本课最重要的设计判断

- Session 是执行容器，不是聊天记录
- Part 是执行过程的表达层
- Processor 是执行事件的收束层
- Agent / Permission / Tool 一起定义行为边界

你只要把这 4 句话真正吃透，后面的二开就开始有抓手了。

## 脱离源码版主讲

如果你完全不看源码，也应该先能讲清这节课的核心结论:

KiloCode 不是“模型回复一句话、UI 显示一段文本”的系统，而是把 AI 编程过程拆成可执行、可记录、可恢复的运行时世界。

在这个世界里:

- Session 是任务容器
- Part 是过程表达
- Processor 是过程收束
- Agent / Permission / Tool 决定行为边界

只要这 4 个判断成立，你再回去看实现时就不会被类型和文件名淹没。

## 关键抽象解剖卡

### 抽象卡 1: Session

它是什么:

一次 AI 编程任务的执行容器。

为什么必须存在:

因为系统要记录的不只是消息，而是整个任务状态。

没有它会怎样:

系统只能留下碎片化消息，无法稳定回放和恢复执行过程。

### 抽象卡 2: Part

它是什么:

消息内部的结构化执行片段。

为什么必须存在:

因为工具、reasoning、patch、step 都不是单一文本。

没有它会怎样:

过程可见性和调试能力会大幅下降。

### 抽象卡 3: SessionProcessor

它是什么:

把流式执行事件持续写回 Session 世界的收束层。

为什么必须存在:

因为 LLM 和工具过程天然是流式、多阶段的。

没有它会怎样:

执行流会分散在多个地方拼接，系统很难维护。

### 抽象卡 4: Agent / Permission / ToolRegistry

它们是什么:

共同定义“现在这个 Agent 到底能做什么”的边界系统。

为什么必须一起存在:

因为 prompt 只能说明意图，权限和工具决定真实执行边界。

没有它们会怎样:

系统要么太弱，要么失控。

## 常见二开场景

### 场景 1: 新增一个工具

你应该先想到:

- ToolRegistry
- Agent 可用范围
- Permission 边界
- 是否产生新的 Session Part

### 场景 2: 增加一个新的 Agent 模式

你应该先想到:

- 这个模式的行为边界是什么
- 它和现有 `code` / `plan` / `ask` 的区别是什么

### 场景 3: 调整工具调用展示

你应该先想到:

- 这首先是 Part 建模问题，还是 UI 渲染问题
