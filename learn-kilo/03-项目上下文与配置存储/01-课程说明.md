# 03 课：项目上下文与配置存储

## 这节课到底在解决什么

第二课让学习者知道了系统是如何从入口和 `serve` 启动成服务端的。  
但此时还有一个非常关键的问题没有解决：

服务端收到请求之后，它怎么知道“当前我服务的是哪个项目、哪个目录、哪套配置”？

这一课就是专门解决这个问题。

它会带学习者理解 4 个核心点：

1. 项目身份是如何被识别的
2. 为什么这个系统要按目录隔离运行时状态
3. 配置为什么会比“小项目的 config.json”复杂得多
4. 为什么数据库和文件存储会同时存在

很多人第一次学这种系统，最容易在这一课翻车。  
原因不是代码太难，而是思维还停留在：

- “一个程序就一套全局状态”
- “配置就是读一个 json”
- “存储就是数据库或者文件二选一”

而当前仓库恰恰在这三点上都不是这种简单模型。

完成本课后，学习者至少要能：

1. 说清楚 [packages/opencode/src/project/project.ts](/d:/Code/kilocode/packages/opencode/src/project/project.ts) 在做什么。

5. 说清楚 [packages/opencode/src/storage/db.ts](/d:/Code/kilocode/packages/opencode/src/storage/db.ts) 和文件存储为什么要共存。

## 这一课明确不追什么

为了避免被复杂度压住，本课刻意不追：

- `Config` 里每一个 schema 字段的具体含义
- 所有 legacy migration 分支的细节
- 数据库 schema 的全部表结构
- 所有 project 更新、图标发现、命令配置等边缘能力

本课只回答：

- “系统如何识别当前项目”
- “系统如何把状态绑定到当前目录”
- “配置与存储为什么会长成现在这样”
