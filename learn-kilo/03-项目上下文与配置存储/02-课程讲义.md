# 讲义

## 一、从“服务端收到请求”到“系统知道我在哪个项目里”

第二课结束时，学习者已经知道：

- 有一个服务端
- 它会处理请求
- 它会通过 `directory` 等信息建立上下文

但这里其实还有一个很深的工程问题：

收到一个目录以后，系统到底怎么判断：

- 这个目录属于哪个项目
- 这个项目的逻辑身份是什么
- 这个目录当前是否在某个 worktree 里
- 这次请求应读哪套配置
- 这次请求应把状态挂在哪里

这一整串问题，就是第三课真正要解决的。

## 二、为什么“当前目录”不能只当成一个字符串

对于很多小程序来说，当前目录只是一个背景参数。

但在当前仓库里，目录的意义要大得多，因为它决定：

- project identity
- worktree 边界
- 配置作用域
- Session 归属
- 工具权限边界

也就是说，在这里“当前目录”不是附属信息，而是运行时语义入口。

所以后面的 `Project` 和 `Instance`，本质上都在围绕这个问题工作：

“怎样把目录变成稳定、可复用、可隔离的系统上下文？”

## 三、`Project` 解决的是“身份识别”问题

请看：

- [packages/opencode/src/project/project.ts](/d:/Code/kilocode/packages/opencode/src/project/project.ts)

很多初学者第一次看这个文件，会以为它只是帮你找到 Git 根目录。  
这只是它做的事情之一，不是它真正重要的地方。

它真正重要的地方是：

- 把目录映射成“逻辑项目身份”

### 为什么不能直接把路径当 project id

因为路径会发生很多变化：

- 你可能在不同的子目录工作
- 你可能进入 worktree
- 你可能把项目放到不同磁盘位置

如果 project id 只是路径，那么：

- 同一个逻辑项目在不同目录下会被当成不同项目
- Session 迁移与共享会变得更乱
- 数据归属会很不稳定

所以 `fromDirectory(...)` 会去做更稳定的身份识别。

### `fromDirectory(...)` 做了什么

第一次读它时，请先只抓主干，不要追所有边缘逻辑。

它大致在做：

1. 向上找 `.git`
2. 判断当前目录是否在 Git 项目里
3. 找 Git root / common-dir / worktree
4. 生成一个更稳定的项目 id
5. 从数据库里查这个项目以前有没有记录
6. 如果没有，就创建一个新的 Project 记录

这意味着：

`Project` 不是一个“路径工具”，而是一个“项目身份解析器”。

## 四、为什么这里会同时出现 `directory`、`sandbox`、`worktree`

这一点特别容易让初学者迷糊。

你第一次看时不用急着精确背定义，但要先建立一个感觉：

- `directory`
  当前请求或当前操作实际发生的位置
- `sandbox`
  某个实际工作目录语义
- `worktree`
  更靠近 Git 工作树边界的逻辑概念

为什么会同时存在这些概念？

因为真实工程里的“工作位置”和“项目身份边界”往往不是一个层次的东西。

如果把它们全压成一个词，后面很多能力都会模糊。

## 五、`Instance` 解决的是“上下文承载”问题

现在看：

- [packages/opencode/src/project/instance.ts](/d:/Code/kilocode/packages/opencode/src/project/instance.ts)

如果 `Project` 负责“识别我是谁”，那么 `Instance` 负责：

- “在这个身份上承载状态和运行时上下文”

这是非常关键的区别。

很多人会下意识想：

- “既然已经识别出 project 了，那直接在全局放一份状态不就行了？”

这就是第三课最重要的误区之一。

### `Instance.provide(...)`

它做的事情不是简单调用回调，而是：

- 以目录为键
- 建立当前上下文
- 让后续逻辑可以通过统一方式访问当前目录、当前 worktree、当前 project

### `Instance.state(...)`

它不是普通缓存，而是：

- 按当前目录分桶的状态工厂

这带来的收益很大：

- 不同目录之间不会串状态
- 状态可以懒加载
- 状态可以统一销毁

这正是大型工程里很常见但小项目里不一定需要的设计。

## 六、为什么全局单例在这里会很危险

请认真理解这一点。

如果你把这个仓库的很多状态都做成普通全局单例，短期内也许“能跑”，但很快会出现一些很难调的问题：

- 配置作用域串了
- 不同目录的 Session 串了
- worktree 行为错乱
- 扩展 / 服务端长生命周期下状态无法可靠清理

所以 `Instance` 不是为了炫技巧，而是为了控制真实复杂度。

你可以把它理解成：

- 这是系统在告诉你：“状态不是全局的，而是有目录边界的。”

## 七、为什么 `Config` 会看起来这么复杂

现在来看：

- [packages/opencode/src/config/config.ts](/d:/Code/kilocode/packages/opencode/src/config/config.ts)

这份文件第一次看几乎所有人都会有反应：

- “怎么这么长？”

这是正常的。

但你要从工程现实去理解它，而不是立刻把它判定成设计糟糕。

### `Config` 复杂的原因不是“坏代码”，而是“真实来源很多”

你在这份文件里会看到很多来源层：

- legacy Kilocode 模式与规则迁移
- 组织模式
- 全局配置
- 项目配置
- `.kilo` / `.opencode` 目录配置
- 托管配置
- 环境变量覆盖

这说明它在面对的是一个成熟产品的真实情况，而不是一份教科书里的简化配置。

### `Config.state`

这一点很关键。

你会发现它也是基于 `Instance.state(...)` 建立的。  
这意味着配置不是“全局唯一值”，而是和当前目录上下文强绑定的运行时视图。

这就解释了为什么：

- 同一个程序在不同目录下，配置体验会不一样

## 八、为什么配置来源多，不等于设计差

这是非常值得建立的工程直觉。

很多初学者会天然觉得：

- “来源越多，越说明这个模块设计不干净”

实际上在成熟产品里，来源层变多往往说明：

- 产品有历史
- 有兼容要求
- 有多环境、多用户、多组织场景
- 有新的能力不断叠加

所以面对复杂配置模块时，你应该先问：

- “这些来源是不是都真的有现实意义？”

而不是先下结论说“这一定是坏设计”。

## 九、为什么数据库和文件存储会同时存在

现在看：

- [packages/opencode/src/storage/db.ts](/d:/Code/kilocode/packages/opencode/src/storage/db.ts)

以及：

- [packages/opencode/src/storage/storage.ts](/d:/Code/kilocode/packages/opencode/src/storage/storage.ts)

你要先建立一个真实工程常识：

存储层很少只有一种形态。

### 数据库适合什么

数据库更适合：

- 核心结构化数据
- 查询需求明确的数据
- 需要一致性的数据

### 文件存储适合什么

文件更适合：

- 某些补充性数据
- 快照式数据
- 导出、缓存、过渡性数据

所以当你看到“数据库 + 文件”并存时，不应该第一反应是混乱，而应该先问：

- 它们是不是各自承担了不同职责？

## 十、这一课真正要带走的结论

请你最后一定要带走下面 4 个结论：

1. `Project` 解决的是“项目身份识别”问题，不是简单路径问题。
2. `Instance` 解决的是“按目录隔离上下文和状态”问题，不是简单缓存问题。
3. `Config` 的复杂性来自真实来源层次，而不是简单的编码失控。
4. 存储分层是现实工程的常态，数据库和文件并存并不奇怪。

如果你能把这 4 个结论讲清楚，这一课就真正起作用了。
