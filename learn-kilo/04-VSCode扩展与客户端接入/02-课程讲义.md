# 04 课程讲义

## 一、`extension.ts` 的本质是装配入口

`packages/kilo-vscode/src/extension.ts` 做的事情很多，但你要把它看成“装配图”。

它负责:

- 创建共享 `KiloConnectionService`
- 注册 sidebar provider 和 tab panel
- 注册 Agent Manager
- 注册 diff viewer、settings editor、sub-agent viewer
- 注册 commands、serializer、autocomplete、browser automation

重点不是细节，而是:

扩展端所有能力都围绕同一个连接层装配。

## 二、`ServerManager` 在做什么

`packages/kilo-vscode/src/services/cli-backend/server-manager.ts` 真正做的是:

- 找到打包好的 CLI
- 生成随机密码
- 设置扩展环境变量
- 启动 `kilo serve --port 0`
- 解析端口
- 管理进程销毁

这里有个关键结论:

扩展端不是在自己实现后端，而是在本地拉起同一个核心运行时。

## 三、为什么 `KiloConnectionService` 必须共享

`packages/kilo-vscode/src/services/cli-backend/connection-service.ts` 管的是:

- 单个 `ServerManager`
- 单个 SDK client
- 单个 SSE adapter
- 多个 provider / panel 的事件广播

为什么必须共享:

- 不同 webview 不应该各自拉起一个 server
- 状态广播必须一致
- 会话和提示拒绝要能跨视图同步
- Agent Manager 需要共享后端世界

## 四、Agent Manager 的工程意义

Agent Manager 不是第二套系统。

它只是把:

- 多 session
- 多 tab
- 多 worktree

编排进同一个连接世界里。

这意味着它的复杂度主要在客户端编排，不在后端重做。

## 五、扩展端最关键的分层判断

### extension host

负责宿主注册、命令、生命周期、创建 provider。

### CLI backend 连接层

负责 server 进程、SDK client、SSE、状态同步。

### webview UI

负责展示与交互，不应该偷偷接管核心业务。

## 本课最重要的设计判断

- 扩展不是系统主体
- 扩展侧的价值是稳定接入核心运行时
- 共享连接是多面板、多会话体验成立的前提

## 脱离源码版主讲

如果你不打开代码，也应该先掌握这节课的主体判断:

VS Code 扩展只是 KiloCode 的宿主接入层。它的任务不是重做运行时，而是把本地 CLI backend 拉起来，再把共享连接和事件流稳定地送到多个视图里。

这节课的关键不是 UI，而是三层职责:

- extension host 负责装配
- 连接层负责 server / SDK / SSE
- webview 负责展示与交互

## 关键抽象解剖卡

### 抽象卡 1: Extension Host

它是什么:

VS Code 环境里的宿主装配层。

为什么必须存在:

因为命令、provider、panel、serializer 都要在这里注册。

没有它会怎样:

扩展能力无法进入 VS Code 生命周期。

### 抽象卡 2: ServerManager

它是什么:

本地 CLI backend 的进程生命周期管理器。

为什么必须存在:

因为扩展需要稳定地拉起、发现、销毁 `kilo serve`。

没有它会怎样:

扩展无法可靠连接统一后端。

### 抽象卡 3: KiloConnectionService

它是什么:

共享 SDK / SSE / 状态广播的连接层。

为什么必须存在:

因为多个 panel、sidebar、Agent Manager 需要共享同一个后端世界。

没有它会怎样:

连接会重复、状态会分裂。

## 常见二开场景

### 场景 1: 给扩展加一个新入口

你应该先想到:

- 这是 command / provider 装配问题
- 还是需要新增 backend 能力

### 场景 2: 多 panel 状态不一致

你应该先想到:

- 共享连接层是否正确广播
- 是否误把状态放在单个视图里

### 场景 3: 扩展看起来没响应

你应该先想到:

- server 有没有拉起
- client 有没有连接
- SSE 有没有收到

而不是先怪 UI。

你的验证目标是:

- 找到装配入口
- 找到 server 进程生命周期
- 找到共享连接和事件广播
