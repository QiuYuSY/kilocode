# 04 课：会话模型与执行循环

## 这节课到底在解决什么

前三课让学习者已经知道：

- 仓库的产品结构是什么
- 核心入口如何启动
- 服务端如何建立上下文
- 项目身份、目录上下文、配置和存储怎样运作

但是到这里，学习者还没有真正碰到这个系统最“像 AI Agent 平台”的那一层。

这一课要进入的，就是整个业务心脏：

- 会话怎么建模
- 消息怎么建模
- 为什么消息还要拆成更细粒度的 Part
- 一次执行是怎样在系统里流动、更新、落库、再被 UI 消费的

如果这一课讲得不好，学习者后面看 Agent、Tool、扩展端时，就会始终有一种模糊感：

- “我知道有 Session，但它到底装了什么？”
- “我知道有工具调用，但为什么 UI 能看见这么细的过程？”
- “我知道有 processor，但它为什么这么复杂？”

这一课就是专门把这些问题打通。

## 本课学习目标

完成本课后，学习者至少要能：

1. 说清楚 [packages/opencode/src/session/index.ts](/d:/Code/kilocode/packages/opencode/src/session/index.ts) 在系统里扮演什么角色。
2. 说清楚为什么 Session 不是聊天记录，而是任务执行容器。
3. 说清楚 [packages/opencode/src/session/message-v2.ts](/d:/Code/kilocode/packages/opencode/src/session/message-v2.ts) 为什么要把 Message 拆成多个 Part。
4. 说清楚 [packages/opencode/src/session/processor.ts](/d:/Code/kilocode/packages/opencode/src/session/processor.ts) 为什么是执行循环核心。
5. 说清楚一次 tool call 从出现、执行、完成到落库的大致路径。

## 这一课明确不追什么

为了让学习者不被代码体量压垮，本课刻意不追：

- 每个 Session 边缘功能，如分享、导出、所有归档逻辑
- token / cost 的全部 provider 细节
- 每个 Part 类型的全部实现分支
- `SessionProcessor` 中每一个 telemetry 或插件扩展点

本课只回答：

- “会话怎么建模”
- “执行怎么流动”
