# 02-课程讲义

## 一、为什么这一课是整个系统最像“AI Agent 平台”的地方

如果前三课主要在讲：

- 系统怎么启动
- 系统怎么成为服务
- 系统怎么识别当前项目和配置

那么这一课开始，学习者终于会碰到最接近“AI 执行本身”的层。

因为无论用户从哪里来：

- CLI
- VS Code 扩展
- Web / Desktop

最终很多关键状态都会落到会话系统里。

也就是说，如果你不知道 Session 是怎么建模的，后面几乎所有“AI 过程感”都会变成黑盒。

## 二、为什么 Session 不是聊天记录

请先看：

- [packages/opencode/src/session/index.ts](/d:/Code/kilocode/packages/opencode/src/session/index.ts)

很多初学者会天然把 Session 想成：

- 用户发一句
- 助手回一句
- 再来一句

这当然不是完全错，但它太浅了。

在当前仓库里，Session 承载的不只是文本消息，它还承载：

- 当前所属 project
- 当前工作目录
- 父子会话关系
- 权限规则
- summary / revert / diff
- 成本与 token 相关信息

这意味着：

Session 在这里更像“任务执行容器”，而不是“对话框容器”。

如果你把它只理解成聊天记录，后面你会看不懂：

- 为什么它要和 project / directory 绑定
- 为什么它有 diff / revert
- 为什么它和工具执行关系这么紧

## 三、为什么 Message 和 Part 要分层建模

现在看：

- [packages/opencode/src/session/message-v2.ts](/d:/Code/kilocode/packages/opencode/src/session/message-v2.ts)

这是整个系统里一个非常值得认真理解的设计点。

### 如果没有 Part，会发生什么

设想一个简化版本：

- Message 只是一整块文本

那你会立刻遇到下面这些表达问题：

- 文本是流式生成的，怎么只更新局部？
- 工具调用开始和结束怎么表示？
- patch 结果怎么表示？
- reasoning 片段怎么表示？
- step-start / step-finish 怎么表示？

所以系统引入了更细粒度的 `Part` 概念。

### Part 的意义

Part 的出现，解决的是：

- 执行过程的可表达性

你会在 `message-v2.ts` 里看到很多类型：

- text
- tool
- step-start
- step-finish
- patch
- reasoning
- file

这说明 Message 不再是一个原子块，而是一个“由多个执行片段组成的结构”。

## 四、为什么 Part 对 UI 和执行层都重要

很多人会以为 Part 是为了 UI 展示方便。

其实它同时服务两层：

### 1. 对执行层

执行过程中产生的不同类型结果，可以分别建模和更新。

### 2. 对展示层

UI 不必等一整个消息都结束，才能知道发生了什么。

所以你在工具面板、diff 展示、流式文本里看到的“细粒度过程感”，背后其实就来自这种数据建模。

## 五、为什么 `SessionProcessor` 必须存在

接下来请看：

- [packages/opencode/src/session/processor.ts](/d:/Code/kilocode/packages/opencode/src/session/processor.ts)

如果 Message / Part 是静态结构，那么 `SessionProcessor` 就是让这些结构真正活起来的执行器。

它的角色不是：

- “做一段业务逻辑”

而是：

- 接收流式执行事件
- 把这些事件映射成会话世界里的状态变化

你可以把它理解成一个“流式事件收束器”。

## 六、如何理解 `SessionProcessor` 的阅读方法

第一次阅读这个文件，不要从第一行一路顺着啃到最后。

正确方式是：

### 先看外层骨架

你会看到：

- `create(...)`
- `process(...)`

这说明它不是随处被直接调用的一段零散逻辑，而是一个有明确输入与生命周期的执行单元。

### 再看三条关键流

#### 1. 文本流

- `text-start`
- `text-delta`
- `text-end`

它解决的是：

- 文本如何逐步进入会话

#### 2. 工具流

- `tool-input-start`
- `tool-call`
- `tool-result`
- `tool-error`

它解决的是：

- 工具调用如何从“即将开始”走到“完成或失败”

#### 3. 步骤结束流

- `finish-step`

它解决的是：

- 本轮步骤结束后，哪些结果需要统一结算和回写

## 七、为什么 `finish-step` 很关键

很多学习者第一次看 `processor.ts`，只会盯着工具调用那几段。

但真正要理解系统为什么完整，就必须认真看 `finish-step`。

因为这一步经常会完成：

- usage 计算
- cost 更新
- patch 落地
- summary 触发
- compaction 判断

这说明执行系统不是“工具跑完就算结束”，而是在不断把运行结果收束为更稳定的会话状态。

## 八、为什么 `SessionProcessor` 不能是普通同步函数

请认真想一个反例：

如果这个系统不用流式 processor，而是：

- 等模型整个回答完
- 等工具全部跑完
- 最后一次性写进数据库

会发生什么？

### 1. UI 过程感大幅下降

你几乎无法优雅地展示流式文本和工具状态变化。

### 2. 中途错误处理会更粗糙

很多中间状态根本来不及被记录。

### 3. 补丁与步骤边界更难表达

系统会更像“事后总结”，而不是“实时执行”。

所以 `SessionProcessor` 必须是事件驱动的，而不是“一次算到底”的同步逻辑。

## 九、为什么这一课会影响你后面理解 Agent 与 Tool

如果这一课不扎实，后面学 Agent 和 Tool 时，学习者会停留在：

- “工具被调用了”
- “模式切换了”

但看不见它们最后怎样落到会话层。

一旦你理解了 Session / Message / Part / Processor，你后面就能把 Agent、Tool、UI 统一理解成：

- 都在往同一个会话世界里写不同类型的状态

这就是这一课真正的价值。

## 十、这一课最重要的 4 个结论

请最后一定要带走：

1. Session 是任务执行容器，不是聊天记录。
2. Part 解决的是执行过程的细粒度表达问题。
3. `SessionProcessor` 是把模型流、工具流、状态流收束到会话系统里的核心执行器。
4. 这个系统之所以有很强的“过程可见性”，靠的不是 UI 技巧，而是底层建模与执行流设计。
