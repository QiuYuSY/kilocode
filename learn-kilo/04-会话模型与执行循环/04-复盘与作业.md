# 04-复盘与作业

## 自测问题

请先独立回答下面 8 个问题：

1. 为什么 Session 不是聊天记录？
2. 为什么 Message 不适合直接做成一整块文本？
3. Part 的存在到底解决了什么问题？
4. 为什么 `ToolPart` 这种结构对执行层很重要？
5. 为什么 `SessionProcessor` 更像流式事件处理器，而不是普通函数？
6. `finish-step` 为什么是执行流中的关键节点？
7. 为什么这个系统能在 UI 里展示很细粒度的执行过程？
8. 如果把当前设计简化成“只有消息，没有 Part，没有流式更新”，最先会损失什么？

## 本课作业

这一课必须交 4 项成果：

### 作业 1：Session 数据模型图

要求：

- 必须包含 Session
- 必须包含 Message
- 必须包含至少 4 类 Part
- 必须说明层级关系

### 作业 2：执行循环时间线

要求：

- 必须包含 `start-step`
- 必须包含文本流
- 必须包含工具流
- 必须包含 `finish-step`

### 作业 3：tool call 生命周期说明

要求：

- 至少写出 pending、running、completed / error 的流动
- 必须说明它最终如何回到会话系统里

### 作业 4：短文说明

要求：

- 主题：为什么 Session 不是聊天记录
- 字数：400 到 600 字
- 不能只写抽象概念

## 验收标准

如果学习者满足下面 4 条，这节课就算通过：

1. 能清楚解释 Session 是任务执行容器。
2. 能清楚解释 Message / Part 分层是为了解决细粒度执行表达问题。
3. 能清楚解释 `SessionProcessor` 是怎样把执行事件写回会话系统的。
4. 能清楚解释为什么这个系统具有强过程可见性。

## 不通过的典型表现

如果学习者出现下面这些情况，说明这一课还没学扎实：

- 还把 Session 理解成聊天框容器
- 还觉得 Part 只是为了 UI 漂亮
- 看不懂工具状态为什么要分 pending / running / completed / error
- 说不清 `SessionProcessor` 为什么存在

## 本课结束后的理想状态

到了这里，学习者还不需要懂所有 Agent 模式和所有 Tool 编排细节。

但必须已经建立一个关键认识：

这个系统真正的业务核心，不是“用户发一句、模型回一句”，而是“执行过程被持续写进一个可观察、可结构化、可复用的会话世界”。

这个认识一旦建立，下一课进入 Agent 和 Tool 编排时，学习者就会更容易理解：

- 模式切换到底在切什么
- 工具调用最终又为什么一定会回到会话系统里
