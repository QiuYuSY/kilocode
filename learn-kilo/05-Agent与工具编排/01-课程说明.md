# 05 课：Agent 与工具编排

## 这节课到底在解决什么

第四课已经让学习者看懂：

- 会话是怎样被建模的
- 执行事件怎样被写回会话系统

但到这里还有两个非常关键的问题没解决：

1. 到底是谁决定“这次任务应该以什么模式运行”？
2. 到底是谁决定“这次任务能调用哪些工具”？

这两个问题，就是第五课的核心。

这一课会带学习者真正理解：

- Agent 模式到底是什么
- 为什么模式切换不只是换一个 prompt
- permission 为什么不是 UI 小弹窗，而是运行时边界
- ToolRegistry 为什么不是一份工具清单，而是动态编排层

如果这一课讲得不扎实，学习者后面去看扩展端、工具面板、自动执行、权限确认时，就会产生很大的误解：

- “模式是不是只是系统提示词不同？”
- “权限不就是点一下 allow / reject 吗？”
- “工具不就是写在数组里的吗？”

这三种理解都太浅。

## 本课学习目标

完成本课后，学习者至少要能：

1. 说清楚 [packages/opencode/src/agent/agent.ts](/d:/Code/kilocode/packages/opencode/src/agent/agent.ts) 在系统里扮演什么角色。
2. 说清楚为什么 `code`、`plan`、`ask`、`debug` 的差异不只是 prompt。
3. 说清楚 [packages/opencode/src/permission/next.ts](/d:/Code/kilocode/packages/opencode/src/permission/next.ts) 为什么属于运行时安全边界。
4. 说清楚 [packages/opencode/src/tool/registry.ts](/d:/Code/kilocode/packages/opencode/src/tool/registry.ts) 为什么是动态编排层。
5. 说清楚模型、配置、插件为什么会影响工具最终暴露给 agent 的集合。

## 这一课明确不追什么

为了让学习者不被细节淹没，本课刻意不追：

- 每个 prompt 文本的完整内容
- 每一个工具实现文件的内部细节
- 每一个 permission 规则分支的全部 edge case
- 所有插件系统的完整机制

本课只回答：

- “模式在切什么”
- “权限在保护什么”
- “工具是怎样被编排出来的”
