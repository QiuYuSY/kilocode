# 02-课程讲义

## 一、为什么这一课要把 Agent、permission、Tool 放在一起讲

很多资料在讲 AI Agent 系统时，会把这三件事分开：

- 模式
- 权限
- 工具

但在真实系统里，它们根本不是三条独立线。

原因很简单：

- 模式决定行为倾向
- 权限决定可跨越的边界
- 工具决定可执行的动作集合

如果你把它们拆开理解，就会出现一种错觉：

- “模式只是语言风格”
- “权限只是确认弹窗”
- “工具只是挂在数组里”

而当前仓库真正的设计恰恰说明：

这三者是在一起定义“agent 能做什么”的。

## 二、为什么 Agent 模式不只是 prompt

请先看：

- [packages/opencode/src/agent/agent.ts](/d:/Code/kilocode/packages/opencode/src/agent/agent.ts)

第一次打开这个文件，很多学习者最先会看到：

- prompt 文本引用
- 模式名字
- description

于是很容易形成一个过浅的理解：

- “模式不就是 prompt 模板吗？”

但如果你再往下看，会发现每个模式还同时携带：

- permission
- mode 类型
- 是否 hidden
- 是否是 subagent
- model / variant 配置
- options

这说明模式真正定义的是：

- 一组行为边界

而 prompt 只是这组边界中的一部分。

### 你最应该先对比哪几个模式

第一次阅读，不要试图理解所有模式。

只盯 4 个就够：

- `code`
- `plan`
- `ask`
- `debug`

这 4 个模式最能代表系统对行为边界的不同切法。

## 三、为什么 `code` 和 `ask` 的差异最值得先看

如果你只能拿一组例子说明“模式不只是 prompt”，那最好的组合就是：

- `code`
- `ask`

### `code`

更接近“完成任务”的主模式。  
它允许更多执行性动作，也允许进入提问、计划等后续流程。

### `ask`

更接近“解释和理解”的模式。  
它在很多动作上明显更保守，尤其在 bash 等可能改动系统状态的能力上更严格。

这说明模式真正控制的是：

- 允许走哪条行为路径

而不是：

- 回答语气有什么不同

## 四、为什么 permission 属于运行时边界

现在请看：

- [packages/opencode/src/permission/next.ts](/d:/Code/kilocode/packages/opencode/src/permission/next.ts)

初学者非常容易把 permission 理解成：

- “一个 UI 弹窗系统”

但如果你认真看这份文件，会发现它做的事情远远不止“弹窗”。

它在做：

- rule 建模
- config 转 ruleset
- ruleset 合并
- 请求 ask / reply
- 运行时 evaluate
- 允许、拒绝、持续允许等状态处理

这说明 permission 不是页面层附属逻辑，而是执行之前必须经过的一道系统边界。

### 为什么它必须是执行层逻辑

因为真正重要的问题不是：

- “页面怎么显示一个确认框”

而是：

- “当前这个动作是否被允许越过某条边界”

这个判断必须发生在工具执行之前。  
所以 permission 的归属天然应该更靠近运行时。

## 五、为什么说 ToolRegistry 是编排中心

现在请看：

- [packages/opencode/src/tool/registry.ts](/d:/Code/kilocode/packages/opencode/src/tool/registry.ts)

很多学习者第一次看这份文件，会觉得它像：

- 一个列工具的地方

这只是最表面的一层。

它真正的价值在于：

- 它决定当前系统上下文下“哪些工具应该被暴露”

请注意这句话的几个关键词：

- 当前系统上下文
- 哪些工具
- 应该被暴露

这说明工具不是默认全量开放，而是经过筛选、拼装、适配后的结果。

## 六、为什么工具不能是写死的数组

如果工具就是一个固定数组，会有什么问题？

### 1. 模型差异无法反映

例如：

- 某些模型更适合 `apply_patch`
- 某些模型更适合 `edit/write`

### 2. 配置差异无法反映

例如实验工具、LSP 工具、batch 工具都可能受配置影响。

### 3. 插件无法优雅接入

如果没有统一汇总层，自定义工具就很难和内置工具并存。

### 4. 权限体系难以协同

工具暴露本身和 permission 边界是相互关联的。

所以 `ToolRegistry` 更像：

- 动态工具编排器

而不是：

- 静态工具表

## 七、为什么 `apply_patch` 和 `edit/write` 会切换

这正是“模型影响工具形态”的一个特别好的例子。

系统不是：

- 工具越多越好

而是：

- 在当前模型语境下，给最合适的工具形态

这说明 ToolRegistry 真正的目标不是“尽量多暴露工具”，而是：

- 让工具与模型能力匹配

这是一种很典型的工程取舍。

## 八、为什么这一课会影响你后面理解扩展端

如果这一课没吃透，后面看扩展端时，学习者很容易把 UI 上看到的东西误解成：

- 一个模式切换器
- 一个工具列表
- 一个确认弹窗

但其实在扩展端显示出来的所有这些东西，背后都依赖：

- Agent 模式的定义
- permission 的运行时判断
- ToolRegistry 的动态编排结果

所以第五课的理解，会直接决定你后面能不能正确理解扩展和 UI。

## 九、这一课真正要带走的 4 个结论

请最后一定要带走：

1. Agent 模式切换的不是提示词风格，而是行为边界。
2. permission 是运行时安全机制，不是 UI 弹窗小功能。
3. ToolRegistry 是动态编排层，不是工具清单。
4. 模型、配置、插件、权限会共同影响“一个 agent 最终能做什么”。
