# 04-复盘与作业

## 自测问题

请先独立回答下面 8 个问题：

1. 为什么 Agent 模式不能被理解成 prompt 模板？
2. `code` 和 `ask` 的本质区别是什么？
3. 为什么 permission 属于运行时边界，而不是 UI 辅助功能？
4. `ask(...)` / `reply(...)` 这类能力说明了 permission 的什么本质？
5. 为什么 ToolRegistry 不是工具清单，而是编排层？
6. 为什么不同模型会影响工具暴露方式？
7. 为什么插件工具需要统一接入层？
8. 如果把当前系统简化成“模式只换 prompt，工具写死，权限只弹窗”，最先会坏掉什么？

## 本课作业

这一课必须交 4 项成果：

### 作业 1：Agent 模式对比表

要求：

- 至少比较 `code`、`plan`、`ask`、`debug`
- 必须包含用途与权限风格

### 作业 2：permission 机制说明

要求：

- 说明规则如何形成
- 说明 ask / reply 机制
- 说明为什么它属于运行时层

### 作业 3：ToolRegistry 说明

要求：

- 说明默认工具、条件工具、插件工具的来源
- 说明模型如何影响工具选择

### 作业 4：短文说明

要求：

- 主题：为什么模式切换不只是 prompt 切换
- 字数：400 到 600 字
- 不能只写抽象结论

## 优秀答案示例结构

### Agent 模式对比表

- 模式名
- 主要用途
- 权限风格
- 行为边界

### permission 机制说明

- 规则从哪里来
- ask / reply 机制
- 为什么它是运行时层

### ToolRegistry 说明

- 默认工具
- 条件工具
- 插件工具
- 模型影响

## 验收标准

如果学习者满足下面 4 条，这节课就算通过：

1. 能清楚解释 Agent 模式是在切换行为边界。
2. 能清楚解释 permission 是运行时安全机制。
3. 能清楚解释 ToolRegistry 是动态编排层。
4. 能清楚解释模型、配置、插件为什么会共同影响工具集合。

## 不通过的典型表现

如果学习者出现下面这些情况，说明这一课还没学扎实：

- 还把模式理解成 prompt 模板
- 还把 permission 理解成 UI 弹窗
- 还把 ToolRegistry 理解成工具数组
- 说不清为什么模型会影响工具暴露方式

## 本课结束后的理想状态

到了这里，学习者还不需要进入扩展端实现细节。

但必须已经建立一个关键认识：

一个 agent 到底能做什么，不是由某一个 prompt 决定的，而是由模式、权限、工具编排共同决定的。

这个认识一旦建立，下一课进入 VS Code 扩展时，学习者就更容易理解为什么 UI 上看到的模式切换、工具能力和权限请求，背后其实都连着核心运行时设计。
