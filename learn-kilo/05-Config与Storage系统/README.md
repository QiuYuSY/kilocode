# 05 课：Config 与 Storage 系统

## 适合谁

- 已经理解上下文隔离，想知道配置和数据如何落地的人

## 前置要求

- 完成 04 课

## 预计时长

- 总时长：2.5 到 3.5 小时
- 推荐分 3 次完成

## 你已经学会了什么

- 你已经知道系统如何识别项目与隔离状态
- 你知道很多行为最终都依赖当前 Instance 上下文

## 这节课新增什么

- 多来源配置系统
- 配置迁移
- 数据库存储与文件存储的分工

## 完成线

- 最低完成线：知道配置不是只读一个文件
- 标准完成线：能解释主要配置来源和数据库角色
- 深入完成线：能说明为什么这里会有迁移、优先级和多种存储方式

## 难度梯度

1. 观察：认配置来源
2. 追链：认 state / global / update
3. 解释：理解优先级与迁移
4. 思考：理解为何数据不全都放数据库或文件

## 源码阅读顺序卡

1. `packages/opencode/src/config/config.ts`
2. 先只看 `state`、`get`、`getGlobal`
3. 再看配置来源合并段
4. `packages/opencode/src/storage/db.ts`
5. `packages/opencode/src/storage/storage.ts`

## 必须关注的函数/结构

- `Config.state`
- `Config.get`
- `Config.getGlobal`
- `Database.Client`
- `Database.use`

## 这课暂时不用深挖

- 每个配置字段的完整 schema 细节
- 所有 legacy migration 的每个分支

## 在真实开发中有什么用

- 帮你定位“为什么这里配置没生效”
- 帮你理解真实工程里为什么总会出现迁移代码
- 帮你判断一个新功能的数据该存数据库还是存文件

## 常见误区

- 以为配置系统只是在读一个 json 文件
- 以为复杂配置来源说明代码设计不好
- 以为数据库层一定能替代全部文件存储

## 本课结论

Config 模块复杂不是因为混乱，而是因为它承担了现实世界中的多来源配置与迁移责任。

## 下一课为什么学它

配置和存储明白之后，下一步就可以进入真正的业务容器：Session 数据模型。
