# 带读讲义

## 这份带读适合什么时候用

- 你第一次打开 `config.ts` 被它吓到的时候
- 你想知道“我第一轮到底该读哪一小部分”

## 阅读前准备

这份文件第一次绝对不要整份线性通读。

第一次阅读只追：

- `state`
- `get`
- `getGlobal`
- `global`
- 数据库存取入口

## 第 1 步：先找 `Config.state`

理解：

- 它为什么也是 `Instance.state(...)`
- 它返回的是什么

### 停下来回答

- 为什么配置读取也要和当前目录绑定？

## 第 2 步：看 `get` 与 `getGlobal`

这一段帮助你先区分：

- 当前实例配置
- 全局配置

### 停下来回答

- 这两层为什么都需要存在？

## 第 3 步：再回去看 `state` 里合并来源的顺序

这时候才开始扫配置来源。

你第一轮只记大类：

- legacy 迁移
- global
- project
- config dir
- managed
- env 覆盖

### 停下来回答

- 为什么真实工程里的配置来源一定会越来越多？

## 第 4 步：跳到 `packages/opencode/src/storage/db.ts`

只看：

- `Database.Path`
- `Client`
- `use`
- `transaction`

### 停下来回答

- 为什么数据库访问也被统一包了一层？

## 第 5 步：再看 `storage.ts`

只感受它和数据库的分工，不用深挖所有实现。

### 停下来回答

- 哪些数据适合结构化落库，哪些更适合文件存储？

## 第一次阅读先停在哪

能分清：

- 配置来源的主要层次
- 数据库存取入口
- 文件存储仍然存在的原因

就够了。

## 读完后的最小输出

画一张简单图：

`Config sources -> merged config -> runtime use`

再写 3 句话解释数据库与文件存储为什么同时存在。
