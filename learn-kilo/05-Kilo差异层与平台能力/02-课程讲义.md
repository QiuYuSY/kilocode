# 05 课程讲义

## 一、Kilo 差异不是“皮肤层”

Kilo 相比 upstream OpenCode 的增加，远不只是:

- 品牌名称
- logo
- 文案

真正的差异主要包括:

- Kilo 鉴权与 provider 路由
- 平台遥测
- 组织模式
- legacy Kilocode 配置迁移
- 扩展侧平台接入
- Kilo 专属后端 routes

## 二、为什么 `index.ts` 和 `server.ts` 里有很多 `kilocode_change`

这些地方是 fork 差异最容易“长进核心主链”的位置。

比如:

- `index.ts` 里接入了 Telemetry 和 legacy auth migration
- `server.ts` 里接入了 Kilo routes、telemetry routes、kilocode routes

这类改动虽然落在共享文件里，但通过 `kilocode_change` 标记，后续 merge upstream 时才有机会快速识别冲突范围。

## 三、`config.ts` 为什么是 Kilo 差异最密集的地方之一

看 `packages/opencode/src/config/config.ts` 时，你会发现它不只是读配置文件。

它还处理了:

- legacy Kilocode modes 迁移
- workflows 迁移
- rules 迁移
- MCP 配置迁移
- `.kilocodeignore` 迁移
- organization modes 拉取
- 多层 config merge

这说明 Kilo 的平台化过程，不是从零开始，而是带着历史演化包袱往前走。

也正因为这样，配置层复杂不是偶然，而是系统演进痕迹。

## 四、Kilo Gateway 的角色

`packages/kilo-gateway/` 不是一个边缘包。

它承接的是:

- Kilo auth
- Provider 封装
- Header 构造
- 模式 API
- 设备授权流程
- 可选的 server routes

这意味着它是 Kilo 平台能力进入核心运行时的重要桥梁。

## 五、Telemetry 的角色

`packages/kilo-telemetry/` 也不是单纯埋点工具。

它承担的是:

- 统一 Telemetry 接口
- identity 处理
- OTel 集成
- 运行过程统计

当你看到 `SessionProcessor` 里会在 `finish-step` 上报 completion 信息时，就应该意识到:

Telemetry 也是执行系统的一部分，而不是页面层装饰。

## 六、为什么 Kilo 专属改动要尽量集中

因为这是 fork 项目。

越多 Kilo 专属代码散落在 upstream 共享文件里，后续 merge 成本越高。

所以原则是:

- 能放 `packages/opencode/src/kilocode/` 就别散在共享目录
- 能放 Kilo 专属包就尽量别改 shared package
- 必须改共享文件时，用 `kilocode_change` 标明边界

## 本课最重要的设计判断

- Kilo 差异层是“平台能力层”，不是“品牌层”
- 配置迁移复杂，反映的是历史兼容成本
- fork 项目最重要的工程能力之一就是控制差异落点

## 脱离源码版主讲

如果你完全不看源码，也应该先能讲出这节课:

Kilo 相对 upstream OpenCode 的差异，不在表面，而在平台能力。

这些差异包括:

- 鉴权
- provider / gateway
- telemetry
- organization modes
- legacy Kilocode 配置迁移

所以 Kilo 差异层的意义，不是展示品牌，而是承接平台化演进。

## 关键抽象解剖卡

### 抽象卡 1: Kilo 差异层

它是什么:

fork 之后 Kilo 专属的平台能力集合。

为什么必须存在:

因为 Kilo 不是纯上游镜像，而是有自己的产品和平台逻辑。

没有它会怎样:

Kilo 专属逻辑会散落在共享代码里。

### 抽象卡 2: Config 迁移层

它是什么:

把 legacy Kilocode 配置和当前配置系统接起来的兼容层。

为什么必须存在:

因为真实产品演进不能假设所有用户都从零开始。

没有它会怎样:

旧用户能力断裂，迁移成本转移到用户。

### 抽象卡 3: Kilo Gateway

它是什么:

Kilo 平台能力进入核心运行时的重要桥梁。

为什么必须存在:

因为鉴权、provider 头部、模式 API 等逻辑不是通用运行时天然自带的。

没有它会怎样:

Kilo 平台逻辑只能硬塞进共享主链。

## 常见二开场景

### 场景 1: 新增 Kilo 专属 header

你应该先想到:

- 这是平台差异层，不是共享运行时层

### 场景 2: 新增 organization modes 能力

你应该先想到:

- Gateway / config / migration 可能都会被影响

### 场景 3: 修改 Kilo telemetry

你应该先想到:

- 这是平台观测能力，不是 UI 文案层
