# 02-课程讲义

## 一、为什么这一课最重要的第一句话是“扩展不是后端”

当学习者第一次进入 [packages/kilo-vscode](/d:/Code/kilocode/packages/kilo-vscode) 时，非常容易出现一种错觉：

- 这里有入口
- 这里有服务
- 这里有 webview
- 这里有 Agent Manager

那是不是这里才是系统主体？

答案是否定的。

你必须先把这一点立住：

VS Code 扩展是重要客户端，但不是核心引擎。

它真正做的是：

- 启动核心后端
- 建立连接
- 接收事件
- 渲染 UI
- 转发用户操作

如果你不先纠正这个直觉，后面你看任何扩展代码，都会把“客户端编排层”误当成“系统执行主体”。

## 二、`extension.ts` 为什么值得先看

请看：

- [packages/kilo-vscode/src/extension.ts](/d:/Code/kilocode/packages/kilo-vscode/src/extension.ts)

这份文件不是“业务实现中心”，而是：

- 扩展端的装配入口

你在这份文件里能看到：

- provider 注册
- command 注册
- serializer 注册
- shared service 初始化
- autocomplete 接入
- backend 连接管理

这说明 `extension.ts` 的主要角色不是执行业务，而是把扩展端各个能力组织起来。

### 为什么它像“装配点”而不是“业务中心”

因为真正的职责大多分散在更具体的服务里，例如：

- `ServerManager`
- `KiloConnectionService`
- `KiloProvider`
- Agent Manager provider

所以看 `extension.ts` 的正确姿势不是：

- 逐行理解一切

而是：

- 看它装配了哪些关键角色

## 三、为什么扩展端需要自己启动 CLI backend

这一点很关键。

扩展不能默认假设用户电脑上已经跑着一个全局、可用、配置正确的后端服务。

所以它必须自己做：

- 找到 bundled CLI binary
- 启动 `kilo serve --port 0`
- 读取真实端口
- 管理生命周期

这就是：

- [packages/kilo-vscode/src/services/cli-backend/server-manager.ts](/d:/Code/kilocode/packages/kilo-vscode/src/services/cli-backend/server-manager.ts)

存在的意义。

### `ServerManager` 真正负责什么

它不是普通工具类，而是：

- 扩展端本地后端进程生命周期管理器

它要做的事情包括：

- 找到 CLI 可执行文件
- 生成 password
- 以合适环境变量启动 `kilo serve`
- 从 stdout 中识别端口
- 处理失败、超时、退出、清理

这说明扩展端不是“顺手调用一下 CLI”，而是在管理一个真实后端进程。

## 四、为什么 `KiloConnectionService` 比普通 client 更重要

现在请看：

- [packages/kilo-vscode/src/services/cli-backend/connection-service.ts](/d:/Code/kilocode/packages/kilo-vscode/src/services/cli-backend/connection-service.ts)

如果你把它理解成“包一层 HTTP 调用”，就太低估它了。

它实际上在统一管理：

- `ServerManager`
- SDK client
- SSE client
- connection state
- 事件广播
- 多 webview 共享连接

这说明它更像：

- 扩展端连接编排中心

而不是：

- 一个请求工具类

### 为什么共享连接非常重要

设想一个反例：

如果每个 webview、每个 tab、每个 panel 都自己连一次后端，会发生什么？

- 重复启动或重复连接
- 状态难同步
- 事件容易分叉
- 多会话环境更容易混乱

所以 `KiloConnectionService` 的存在，本质上是在帮扩展端建立“共享且稳定的连接世界”。

## 五、为什么 webview 不能被当成普通前端应用

请看：

- [packages/kilo-vscode/webview-ui/src/App.tsx](/d:/Code/kilocode/packages/kilo-vscode/webview-ui/src/App.tsx)

很多学习者一看到 Solid 组件树，会下意识用普通前端项目的眼光去看它。

这样会忽略一个关键事实：

webview 运行在 VS Code 的扩展容器环境里，它不是一个独立掌控一切的前端应用。

它做不了很多必须由 extension host 做的事，例如：

- 拉起本地进程
- 管理 VS Code 原生命令
- 打开文件
- 维护共享后端连接

所以这套架构真正关键的不是“谁渲染 UI”，而是：

- extension host 和 webview 之间如何分工

### 这两层该怎么理解

#### extension host

更接近：

- 系统能力承接层

#### webview

更接近：

- UI 展示与交互层

这不是语法差异，而是责任边界差异。

## 六、为什么 Agent Manager 不是另一套系统

到了这里，最容易让学习者误判的就是 Agent Manager。

因为它看起来像一个很大的功能块：

- 多会话
- 多 tab
- worktree
- panel

于是很多人会天然觉得：

- “它是不是另一套后端？”

答案是否定的。

Agent Manager 之所以特别，恰恰是因为：

- 它在同一个后端之上，做多会话编排

这意味着：

- 会话隔离主要通过目录和 worktree 达成
- backend 连接不是每个会话独立一份
- 资源使用和状态管理都更可控

这是一种很典型的工程平衡：

- 既要支持多会话
- 又不能让每个会话都各起一套后端

## 七、为什么这一课会直接影响你对 UI 的理解

如果这一课没吃透，学习者在看扩展端 UI 时会不断产生浅层判断：

- “这里有个模式切换按钮”
- “这里有个工具面板”
- “这里有个会话页签”

但无法解释：

- 它们的数据从哪里来
- 它们的状态为什么能共享
- 它们为什么能在多面板之间一致

而一旦你理解了：

- `ServerManager`
- `KiloConnectionService`
- extension host / webview 边界

你就会开始把扩展端 UI 理解成：

- 一个建立在共享后端连接和共享事件流之上的展示层

这会让你后面看前端与 SDK 时顺很多。

## 八、这一课真正要带走的 4 个结论

请最后一定要带走：

1. VS Code 扩展是客户端，不是系统主体。
2. `ServerManager` 负责本地 CLI backend 的生命周期。
3. `KiloConnectionService` 负责连接、事件和多面板共享状态。
4. Agent Manager 是共享后端上的多会话编排能力，不是另一套系统。
