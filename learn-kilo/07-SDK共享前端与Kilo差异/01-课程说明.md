# 07 课：SDK、共享前端与 Kilo 差异

## 这节课到底在解决什么

第六课已经让学习者知道：

- 扩展不是系统主体
- 扩展只是客户端
- 客户端通过共享后端来工作

但此时还会剩下 3 个很关键的问题：

1. 客户端和后端之间，为什么还需要单独一层 SDK？
2. 既然已经有多个客户端，为什么还要做共享前端和组件层？
3. Kilo 相比 upstream OpenCode，究竟多加了哪些真正重要的能力？

这三个问题看起来分散，但其实它们都在回答同一个更大的问题：

- “一个成熟的多客户端系统，除了核心引擎和客户端本身，还需要哪些桥梁层与增量层？”

第七课就是专门把这个问题讲清楚。

如果这一课讲不好，学习者很容易形成几种典型误解：

- “SDK 不就是方便封装一下请求吗？”
- “共享前端不就是一套 UI 搬来搬去吗？”
- “Kilo 差异不就是加了一些品牌和接口？”

这些理解都太浅。

## 本课学习目标

完成本课后，学习者至少要能：

1. 说清楚 [packages/sdk/js](/d:/Code/kilocode/packages/sdk/js) 为什么是契约层，而不只是一个方便库。
2. 说清楚 [packages/app](/d:/Code/kilocode/packages/app)、[packages/kilo-vscode/webview-ui](/d:/Code/kilocode/packages/kilo-vscode/webview-ui)、[packages/kilo-ui](/d:/Code/kilocode/packages/kilo-ui) 三者的关系。
3. 说清楚共享前端不等于“所有客户端完全共用同一套页面”。
4. 说清楚 [packages/kilo-gateway](/d:/Code/kilocode/packages/kilo-gateway)、[packages/kilo-telemetry](/d:/Code/kilocode/packages/kilo-telemetry)、[packages/opencode/src/kilocode](/d:/Code/kilocode/packages/opencode/src/kilocode) 分别代表什么样的 Kilo 增量能力。
5. 说清楚为什么 Kilo / upstream 差异必须保持可识别。

## 建议学习时长

- 预估总时长：4 到 5 小时
- 推荐拆成 4 段：
  1. 50 分钟读 SDK 层
  2. 60 分钟读共享前端与组件层
  3. 50 分钟读 Kilo 增量层
  4. 40 到 50 分钟做实验和复盘

## 这一课明确不追什么

为了让学习者不被范围拖散，本课刻意不追：

- SDK 生成代码的全部细节
- 所有前端组件的实现细节
- Kilo Gateway 的每个 API 细节
- Telemetry 的每个埋点细节

本课只回答：

- “SDK 为什么是桥梁”
- “共享前端到底共享了什么”
- “Kilo 增量层在系统里处于什么地位”
