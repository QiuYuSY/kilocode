# 02-课程讲义

## 一、为什么这一课其实是在讲“桥梁层”

到目前为止，学习者已经见过两层：

- 核心引擎层
- 客户端层

但一个成熟系统如果只有这两层，通常会很痛苦：

- 每个客户端都要自己和后端对接
- 每个客户端都要自己决定界面表达方式
- fork 差异会慢慢散落到各处，越来越难辨认

所以真实工程里通常会出现“桥梁层”和“增量层”。

在当前仓库里，这些层主要体现为：

- SDK
- 共享前端
- 组件层
- Kilo-specific 包与目录

这也是为什么第七课看起来像在讲几个散点，但其实是在讲同一件事：

- “如何让多客户端系统保持一致性、复用性和可演化性”

## 二、为什么 SDK 不是“可有可无的请求封装”

现在先看：

- [packages/sdk/js/src/client.ts](/d:/Code/kilocode/packages/sdk/js/src/client.ts)

很多初学者看到 SDK 时，第一反应很自然：

- “这不就是把 fetch 包一层吗？”

如果项目只有一个小前端，这种想法也许问题不大。  
但在当前仓库里，这个判断太浅。

### SDK 在这里真正承担什么

它承担的是：

- 客户端与服务端之间的统一调用契约

请注意这个词：

- 契约

它和“方便封装”最大的区别在于：

- 方便封装偏实现习惯
- 契约层偏系统边界

当前仓库里有多个客户端：

- VS Code 扩展
- app / Web / Desktop

如果没有 SDK，每个客户端都要自己决定：

- 接口怎么调
- 参数怎么传
- header 怎么补
- streaming 怎么接

这样很容易越走越散。

### `createKiloClient` 为什么值得看

在 [packages/sdk/js/src/client.ts](/d:/Code/kilocode/packages/sdk/js/src/client.ts) 里，学习者会看到：

- client 的创建
- fetch 行为的兼容处理
- directory 头的注入
- duplex 处理

这说明 SDK 不是在做“语法糖”，而是在做环境适配与接口统一。

所以这一层的真正价值是：

- 让多个客户端站在同一个后端契约之上说话

## 三、为什么共享前端不是“把页面复制来复制去”

现在看：

- [packages/app/src/entry.tsx](/d:/Code/kilocode/packages/app/src/entry.tsx)

以及：

- [packages/kilo-vscode/webview-ui](/d:/Code/kilocode/packages/kilo-vscode/webview-ui)

很多学习者会以为：

- 既然都有前端，那共享前端就应该是一套页面 everywhere

这个理解也不够准确。

当前仓库里真正共享的，不是“所有页面逐文件完全一致”，而是：

- 一部分界面组织方式
- 一部分上下文模式
- 一部分组件体系

### 为什么 `app` 和 `webview-ui` 不会完全一样

因为它们处在不同平台环境：

#### `packages/app`

更接近完整应用环境，服务于 Web / Desktop。

#### `packages/kilo-vscode/webview-ui`

运行在 VS Code webview 环境里，有很强的平台约束。

这意味着共享前端的重点不是“把一切做成完全一样”，而是：

- 在合理边界内复用

这是一种很现实的工程平衡。

## 四、为什么 `kilo-ui` 不是“样式仓库”

现在看：

- [packages/kilo-ui/package.json](/d:/Code/kilocode/packages/kilo-ui/package.json)

第一次看它时，很多人会本能地把组件库理解成：

- “放一些按钮、卡片、样式”

但在这个仓库里，组件库更像：

- 多客户端共享 UI 语言的基础设施

为什么这么说？

因为它不仅提供组件，还提供：

- 主题
- context
- 样式入口
- 组件导出边界

这意味着它的价值是：

- 统一表达方式
- 降低多客户端重复造 UI 的成本
- 帮不同产品保持更一致的界面语义

所以在工程层面，它不是“资源库”，而是“界面基础设施”。

## 五、为什么要单独建立 Kilo 增量视角

现在看：

- [packages/kilo-gateway/src/index.ts](/d:/Code/kilocode/packages/kilo-gateway/src/index.ts)
- [packages/kilo-telemetry/src/index.ts](/d:/Code/kilocode/packages/kilo-telemetry/src/index.ts)
- [packages/opencode/src/kilocode](/d:/Code/kilocode/packages/opencode/src/kilocode)

学习到这一步，学习者已经知道当前仓库不是从零完全独立长出来的，而是带有 upstream 背景。

这时必须建立一个新的阅读视角：

- 哪些是通用核心能力
- 哪些是 Kilo 自己加上的能力

如果没有这层视角，学习者就会把所有代码混成一团：

- 不知道哪些是 upstream 思路
- 不知道哪些是 Kilo 平台差异
- 不知道未来改动应该更偏共享层还是更偏 Kilo-specific 层

## 六、Kilo 增量主要体现在哪些地方

### `packages/kilo-gateway`

更接近：

- 认证
- Kilo API
- 路由与 provider 相关平台能力

### `packages/kilo-telemetry`

更接近：

- 遥测
- 观测
- tracing

### `packages/opencode/src/kilocode`

更接近：

- 在核心运行时里加的 Kilo-specific 能力

这三类东西加起来，构成了“不是 upstream 原样，而是 Kilo 平台版本”的那部分身份。

## 七、为什么 fork 差异必须可识别

这是一个很容易被忽略、但非常工程化的点。

如果一个 fork 项目把自己的差异完全散在共享核心里，不做任何边界控制，后果通常会是：

- 上游同步越来越难
- 冲突越来越多
- 团队越来越难说清“哪些是通用能力，哪些是平台差异”

所以你会看到：

- `kilocode_change`
- `packages/opencode/src/kilocode`
- 单独的 Kilo-specific 包

这些并不只是命名习惯，它们是在帮助系统维持“差异可识别性”。

## 八、为什么这一课会影响你后面做真实改动

如果这一课没学透，学习者后面做小改动时会很容易迷茫：

- 这个功能应该改 SDK 还是改客户端？
- 这个问题应该落在共享前端还是 VS Code webview？
- 这个逻辑是通用核心，还是 Kilo-specific 增量？

而如果这一课扎实了，你会更容易形成这种判断：

- 这是契约层问题
- 这是共享 UI 层问题
- 这是 Kilo 平台能力问题

这会极大降低后面乱改文件的概率。

## 九、这一课真正要带走的 4 个结论

请最后一定要带走：

1. SDK 是客户端与后端之间的契约层，不是普通封装层。
2. 共享前端不是完全统一 UI，而是在平台差异前提下的有边界复用。
3. `kilo-ui` 是多客户端共享界面语言的基础设施。
4. Kilo-specific 能力必须与 upstream 核心保持可识别边界。
