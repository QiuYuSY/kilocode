# 04-复盘与作业

## 自测问题

请先独立回答下面 8 个问题：

1. 为什么 SDK 更像契约层，而不是普通请求封装？
2. 如果没有 SDK，多客户端会遇到什么问题？
3. 为什么共享前端不等于“所有客户端完全共用同一套页面”？
4. `kilo-ui` 为什么是工程基础设施？
5. 为什么 Kilo-specific 能力不能完全散落在通用核心里？
6. `kilo-gateway` 和 `kilo-telemetry` 更偏平台层还是产品层？为什么？
7. 为什么 fork 差异必须保持可识别？
8. 如果把 SDK、共享前端层、Kilo 增量层都删掉，系统最先会在哪些方面变糟？

## 本课作业

这一课必须交 4 项成果：

### 作业 1：server / sdk / client 三层关系图

要求：

- 必须明确标出 SDK 这一层
- 必须说明三层职责区别

### 作业 2：共享前端关系图

要求：

- 必须包含 `packages/app`
- 必须包含 `packages/kilo-vscode/webview-ui`
- 必须包含 `packages/kilo-ui`
- 必须说明共享边界

### 作业 3：Kilo 增量能力清单

要求：

- 至少列出 5 项
- 每项都说明它更偏哪一层

### 作业 4：短文说明

要求：

- 主题：为什么 SDK 是契约层
- 字数：400 到 600 字
- 不能只说“这样更方便”

## 验收标准

如果学习者满足下面 4 条，这节课就算通过：

1. 能清楚解释 SDK 为什么属于桥梁层 / 契约层。
2. 能清楚解释共享前端是在复用什么，而不是简单复制页面。
3. 能清楚解释 `kilo-ui` 的工程角色。
4. 能清楚解释 Kilo 与 upstream 差异为什么必须可识别。

## 不通过的典型表现

如果学习者出现下面这些情况，说明这一课还没学扎实：

- 还把 SDK 理解成方便工具
- 还把共享前端理解成“页面全部一样”
- 还把 `kilo-ui` 理解成纯样式资产
- 说不清哪些是 Kilo 平台增量

## 本课结束后的理想状态

到了这里，学习者已经不仅能理解核心引擎和客户端，也开始理解真正的大型工程为什么会长出桥梁层与增量层。

这个状态非常重要，因为到下一课进入毕业项目时，学习者已经具备一种更成熟的判断：

- 这个问题是核心引擎问题
- 这个问题是客户端问题
- 这个问题是契约层问题
- 这个问题是 Kilo 平台特有问题

一旦这种判断能力建立起来，后面的实际输出质量会高很多。
