# 07 课程讲义

## 一、验证不是“最后有空再补”

在 KiloCode 里，验证链路要跟着改动类型走。

### 改核心运行时

至少考虑:

- typecheck
- `packages/opencode/` 下的测试
- 相关命令或 server 行为的手动验证

### 改 route / server endpoint

除了 typecheck 和测试，还要考虑:

- SDK 是否需要 regen
- 客户端是否需要同步适配

### 改扩展端

至少考虑:

- typecheck
- `bun run extension`
- 对应交互路径的手动验证

### 改文档链接相关

要考虑:

- source links 提取命令

## 二、调试时优先抓主链日志

不管是 CLI 还是扩展，最重要的是先定位主链断在哪。

例如扩展侧问题，优先排查:

- `ServerManager` 有没有成功拉起 CLI
- 端口有没有拿到
- `KiloConnectionService` 有没有连上
- SSE 有没有正常工作

而不是一上来先怀疑 UI。

## 三、fork 项目特有的交付要求

### 1. 控制共享文件改动

能不改共享文件就不改。

### 2. 必要时加 `kilocode_change`

尤其是改 upstream 共享文件时。

### 3. 注意 source links 和 SDK 生成

这些都不是“可选流程”，CI 会看。

### 4. 不要把 Kilo 专属能力随意塞进共享层

这会提高后续 merge 成本。

## 四、最小验证清单怎么写

一个健康的小改动，至少应该有:

- 改动范围说明
- 影响面说明
- 运行验证
- 自动化验证
- 特殊生成步骤

## 本课最重要的设计判断

交付能力的核心，不是把所有命令都跑一遍，而是知道这次改动真正需要验证什么。

## 脱离源码版主讲

如果你不看源码，也应该先学会这节课最重要的交付观:

不同类型的改动，验证方式不一样。

你不需要机械地把所有命令全跑一遍，但你必须能准确回答:

- 这次改动应该验证哪条链
- 这次改动会不会影响 SDK
- 这次改动会不会触发 Kilo fork 约束

这才叫真正的交付能力。

## 关键抽象解剖卡

### 抽象卡 1: 最小验证链

它是什么:

和当前改动最相关的一组最小验证动作。

为什么必须存在:

因为验证也需要边界感。

没有它会怎样:

不是乱跑命令，就是漏掉关键验证。

### 抽象卡 2: 主链排查顺序

它是什么:

出现问题时，按运行主链从前往后排查的顺序。

为什么必须存在:

因为很多表面问题都不发生在表面。

没有它会怎样:

你会先怪 UI、先怪体验，却错过真正的故障点。

### 抽象卡 3: 交付约束

它是什么:

fork 项目里除了功能本身之外必须满足的生成、检查和边界规则。

为什么必须存在:

因为 CI、SDK、source links、`kilocode_change` 都是真实交付的一部分。

没有它会怎样:

改动能跑，但无法稳定提交和维护。

## 常见二开场景

### 场景 1: route 改了

你应该先想到:

- SDK 是否需要 regen
- 客户端是否同步适配

### 场景 2: 扩展端连不上 backend

你应该先想到:

- server 是否成功拉起
- 端口是否被解析
- 共享连接是否建立

### 场景 3: 改了共享文件里的 Kilo 差异

你应该先想到:

- 是否已经清楚标识差异边界

## 源码映射附录

先把这节课的验证和排查方法学会，再回到这些文件和命令验证:

- 根 `AGENTS.md`
- `packages/opencode/AGENTS.md`
- `packages/kilo-vscode/src/extension.ts`

你的验证目标是:

- 把课程里的验证清单和真实仓库命令一一对应起来
