# 02-课程讲义

## 一、为什么毕业课不是形式主义

很多人学工程课时，最容易把“毕业作业”当作最后的形式性步骤。

但在真正的大型工程学习里，毕业课的价值恰恰非常高。

原因很简单：

如果没有最后一次完整输出，学习者非常容易停留在：

- 看过源码
- 听懂了一些解释
- 好像知道几个核心概念

但这些都不等于真正掌握。

真正更接近掌握的状态是：

- 你能控制一个清晰范围
- 你能把它讲给别人听
- 你能给出真实文件和真实链路
- 你能自己判断哪些地方还薄弱

毕业课的真正作用，就是把“理解感”变成“输出能力”。

## 二、为什么选题范围控制是毕业课的第一关

大多数学习者在毕业课最先翻车的地方，不是能力不够，而是题目太大。

最常见的失败选题是：

- “讲完整个仓库”
- “讲所有 Agent 模式”
- “讲 VS Code 扩展的全部实现”
- “讲 Kilo 的所有能力”

这些题目有两个问题：

1. 范围太大，最后只能流于概述
2. 证据链太散，最后很难讲深

### 好题目应该是什么样

一个好的毕业题目通常具备三个特点：

#### 1. 范围有限

它最好只围绕一条清晰链路，或一个清晰模块。

#### 2. 能闭环

你能从入口讲到结果，而不是讲到一半就必须跳很多无关支线。

#### 3. 有真实文件落点

你能明确指出：

- 入口文件
- 关键模块
- 关键数据结构

## 三、适合这套仓库的毕业题目类型

### 类型 A：链路讲解型

这是最推荐小白做的类型，因为风险最低，也最能锻炼理解力。

适合题目例如：

- `kilo serve` 从启动到服务成型
- Session / Message / Part 的模型设计
- 一次 tool call 的生命周期
- VS Code 扩展如何连接 CLI backend

这些题目都有共同特点：

- 范围可控
- 主链路明确
- 能和课程内容直接对上

### 类型 B：小型真实改动型

适合已经比较有把握、愿意动手的人。

适合范围例如：

- 一处文案
- 一处日志
- 一处空状态
- 一处轻量 bug
- 一处说明型配置增强

注意：

毕业课不是让你证明自己能改大功能，而是让你证明自己知道如何在理解基础上做出谨慎的真实改动。

## 四、为什么“证据链”比“想法”更重要

毕业成果最怕的不是字少，而是空。

什么叫空？

就是你写了一大段理解，但别人问你：

- “这段结论对应哪个文件？”
- “这个判断对应哪条调用链？”
- “你是从哪里看出来的？”

你答不上来。

所以毕业课一定要有证据链。

一份合格的证据链通常至少包含：

- 入口文件
- 关键模块
- 关键函数 / 结构
- 你引用这些东西的原因

## 五、源码讲解稿到底该长什么样

很多学习者一听“写讲解稿”，会陷入两个极端：

### 极端 1：写成大纲

只有标题，没有解释。

### 极端 2：写成长抄录

把源码逻辑顺着抄一遍，但没有自己的组织。

正确做法是：

### 1. 先给背景

这个问题在系统里处于哪一层？

### 2. 再给入口

从哪个文件、哪个函数开始？

### 3. 再讲主链路

沿着调用和数据流讲。

### 4. 再讲设计点

为什么要这样设计，而不是只讲“发生了什么”。

### 5. 最后给自己的理解

你觉得它最漂亮的点是什么，最难懂的点是什么。

## 六、小型真实改动该如何控制风险

如果选择做改动型毕业项目，一定要记住：

- 你不是来证明自己会改大功能的
- 你是来证明自己知道如何在理解基础上做出可信的小改动

所以你要避免：

- 改自己还没读懂的模块
- 一上来改多个文件、多个层次
- 做一个会影响很多路径的大改动

更稳的方式是：

- 先选一个你已经读过的模块
- 选一个非常小的问题
- 先写清楚问题与证据
- 再改动
- 最后再验证

## 七、为什么复盘在毕业课里非常重要

很多学习者写复盘时会敷衍成：

- 我学到了很多
- 我还要继续努力

这种复盘没有价值。

真正有效的复盘应该至少回答：

1. 我现在最能讲清的 3 个点是什么？
2. 我最薄弱的 3 个点是什么？
3. 如果再给我 2 周，我最该补哪一层？

这会让毕业课不只是交差，而是成为继续深入的起点。

## 八、这一课真正要带走的 4 个结论

请最后一定要带走：

1. 毕业课最重要的不是“写很多”，而是“范围清楚、证据链完整”。
2. 好的毕业题目应该小而完整，而不是大而空。
3. 讲解型与改动型成果都必须建立在真实源码证据之上。
4. 复盘不是附属物，而是把这套课程转成长期能力的重要一步。
