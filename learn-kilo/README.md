# Learn Kilo

这套 `learn-kilo` 不再把目标设为“带你逛完仓库”，而是直接对齐一个更严格的结果:

看完之后，你应该具备对 KiloCode 做二次开发的能力。

这里的“二次开发能力”不是指你记住了几个目录名，而是你真的能做下面这些事:

- 说清 KiloCode 的系统分层，而不是把它误解成一个单纯的 VS Code 插件
- 顺着真实主链解释一次请求如何从入口进入服务端、会话、Agent、Tool、客户端
- 判断一个需求应该改核心运行时、扩展接入层、SDK 契约层，还是 Kilo 的 fork 差异层
- 为一个新能力找到正确落点，控制改动边界，并知道改完后如何验证
- 理解为什么 Kilo 在 upstream OpenCode 基础上增加了 Gateway、Telemetry、legacy 迁移、扩展编排等平台能力

## 这次重构后的课程原则

旧问题我认同:

- 阶段很多，但每一段都太薄
- 讲“看哪里”比讲“为什么这么设计”更多
- 缺少对真实代码主链的剖开
- 缺少“我要在哪里改”的训练
- 看完后只能复述概览，不能做二开

所以新的课程只坚持 5 个原则:

1. 先抓主链，再读细节
2. 先拆设计，再谈实现
3. 每一课都必须落到真实文件和真实改造点
4. 每一课都必须产出可检验结果
5. 课程本身先讲透工程，源码只做附录和证据链

## 学完后的能力标准

完成全部课程后，你至少应该达到下面这条线:

### 能分析

- 能画出 KiloCode 的核心架构图
- 能解释 `packages/opencode`、`packages/kilo-vscode`、`packages/sdk/js`、`packages/kilo-gateway` 的职责边界
- 能说清 `Instance`、`SessionProcessor`、`ToolRegistry`、`KiloConnectionService` 这类抽象存在的原因

### 能定位

- 新增 CLI/Server 能力时知道先看哪里
- 新增扩展端功能时知道 extension host、CLI backend、webview 各改哪里
- 发现一个能力属于 fork 差异层时，知道优先放到 `packages/opencode/src/kilocode/` 或 Kilo 专属包

### 能改

- 能做一个小型工具或 Agent 行为调整
- 能加一个后端 route 并判断是否需要 regen SDK
- 能做一个扩展端接入改动并验证从 CLI backend 到 webview 的整条链

### 能交付

- 知道这个仓库的关键命令、测试入口、SDK 生成、source links 和 merge 友好约束
- 知道什么时候必须加 `kilocode_change`
- 知道如何把改动控制在更容易和 upstream 合并的边界里

## 推荐学习顺序

请不要再把它当“文档浏览”。

推荐顺序是:

1. 先读 [KiloCode架构总册.md](./KiloCode架构总册.md)
2. 再读 [KiloCode二开手册.md](./KiloCode二开手册.md)
3. 再完成 `00-训练营导读`
4. 再学 `01-05`，把总册拆成稳定理解
5. 再学 `06-07`，把手册变成真实工作流
6. 最后做 `08-毕业项目`

## 这套课最重要的变化

以前的重点更像:

- 你看过哪些文件
- 你知道哪些概念

现在的重点改成:

- 你能不能不看源码也先把工程讲透
- 你能不能用源码证据验证自己的设计理解
- 你能不能说出一个需求应该改哪里
- 你能不能在不破坏系统边界的情况下做出改动
