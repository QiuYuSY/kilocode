# 课程升级任务清单

这份文档不再记录“要不要继续补一些导学材料”，而是记录这次课程重构到底改了什么。

## 这次重构解决的核心问题

旧版本的主要问题是:

- 结构看起来完整，但很多课包实际上是空的
- 课程更像源码观光，而不是二次开发训练
- 对 KiloCode 的设计剖析不够，尤其是主链、抽象边界、fork 差异层讲得太薄
- 学完以后知道“在哪”，但不知道“为什么这么设计”和“我应该怎么改”

## 新版课程的重构目标

新版课程必须做到:

- 让学习者能解释 KiloCode 的核心架构，而不是只会背目录
- 让学习者知道一个需求应该改哪一层
- 让学习者真正追过 `index -> serve -> server -> instance -> session -> agent -> tool`
- 让学习者真正理解扩展端只是客户端，不是系统主体
- 让学习者知道 Kilo 特有能力落在哪里，以及为什么要尽量和 upstream 保持低冲突边界
- 让课程最后能落到一个真实的二开项目

## 这次重构的具体变化

### 1. 重构课程主线

从“阶段很多、每段都薄”改成:

- 模块 A: 地图与主链
- 模块 B: 核心设计剖析
- 模块 C: 二次开发工作流
- 模块 D: 毕业项目

### 2. 重构课程目标

从“看过源码”改成:

- 能分析
- 能定位
- 能改
- 能交付

### 3. 重构课包设计

每节课都必须包含:

- 课程说明
- 课程讲义
- 实验与练习
- 复盘与作业
- 课后总复盘模板

并且每节课都必须落在真实文件、真实链路和真实输出物上。

### 4. 增加设计剖析比例

新版更强调:

- 抽象为什么存在
- 分层为什么这样切
- 这个需求为什么不应该改在另一个地方
- Kilo 和 upstream 的差异到底体现在什么层

### 5. 增加二开导向内容

新版加入了:

- 常见需求到改动落点的映射
- SDK regen、source links、typecheck、test 等真实交付要求
- `kilocode_change` 与 merge 友好策略
- 毕业项目里的真实改动型题目

## 完成这次重构后，课程应该达到的标准

如果课程达标，学习者完成后至少应该能:

- 讲明白 `packages/opencode` 为什么是核心
- 讲明白 `packages/kilo-vscode` 和 `packages/sdk/js` 的边界
- 解释 `Instance`、`SessionProcessor`、`ToolRegistry`、`KiloConnectionService` 的设计意义
- 为一个新能力判断正确落点
- 在真实仓库里完成一个可验证的小型二开改动

## 仍然允许继续增强的方向

这次重构已经把课程从“观光型”改成了“训练型”，但以后还可以继续增强:

- 为每节课增加逐段带读版
- 为毕业项目增加标准答案对照
- 为关键链路补动图或时序图
- 为不同角色增加快速路径

但这些都建立在一个前提上:

主课程本身必须先真的能把人带到“能做二开”的程度。
